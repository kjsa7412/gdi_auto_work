# v5 → gdi_auto_work 이식 작업지시서 v1

## 작업 목적
이번 작업의 목적은 `auto_gdi_work_v5`에 누적된 재사용 자산을 `gdi_auto_work`로 이식하여, 새 시스템의 analyze 품질, skeleton 선택 정확도, framework 강제 규칙, runtime 통제 수준을 끌어올리는 것이다.

이번 단계는 **기능 개발**이 아니라 **자산 이식 및 구조 반영** 단계이다.  
즉, v5의 자산을 그대로 복제하는 것이 아니라, 새 시스템의 현재 구조(`.claude / system / work / archive`)에 맞게 **복사 / 병합 / 재작성 / 보류**로 구분하여 반영하라.

- C:\auto_gdi_work_v5

`gdi_auto_work`는 `.claude`, `system`, `work` 중심 구조와 single-active workspace, skeleton 기반 생성, archive 입력 금지 원칙을 채택하고 있다. citeturn433391view0turn433391view1

## 절대 규칙
1. v5 자산을 무조건 복사하지 말고, 반드시 `gdi_auto_work`의 현재 구조와 충돌 여부를 먼저 확인하라.
2. 경로 체계가 다른 자산은 그대로 복사하지 말고 재작성하라.
3. `user/`, `system/runtime/tasks/{taskId}` 등 v5 전용 구조를 `gdi_auto_work`에 그대로 들여오지 말아라.
4. `gdi_auto_work`의 canonical 구조는 `work/task`, `work/fix`, `archive/work_YYYYMMDD_HHMMSS`, `system/*` 이다. 이 구조를 깨는 이식은 금지한다.
5. v5의 analyze 자산은 우선순위가 가장 높다. 특히 PPT 추출, 태그 분류, 화면유형 판정 규칙은 먼저 반영하라.
6. screen-type 명칭은 `gdi_auto_work` 기준을 우선한다.
   - `grid-only` → `list`
   - `search-grid-form` → `list-detail`
   - `form-detail` → `form`
   - `popup-grid` → `popup`
7. dual-spec 구조는 유지하라.
   - 사람 검토용: `people_spec.md`
   - 기계 입력용: `machine_spec.yml`
8. archive는 history only 영역이며 어떤 이식 작업도 archive를 실행 입력으로 사용하게 만들어서는 안 된다.
9. DB 접속정보, 로컬 절대경로, 민감정보는 이식 금지다.
10. popup 파라미터 규칙, single-spec/dual-spec 충돌, 경로 체계 충돌 같은 상충 규칙은 발견 즉시 “재작성” 대상으로 분류하라.
11. 이번 단계에서는 `/test` 체계나 Playwright는 필수 이식 대상이 아니다. analyze/build/fix/clean 중심 구조에 직접 필요한 자산을 우선 반영하라.
12. 이식 결과는 반드시 파일별 액션 상태와 근거를 남겨라.

## 액션 분류 기준

### 1. 복사
다음 조건을 만족하면 **복사** 대상으로 분류하라.
- 현재 `gdi_auto_work`에 동일 목적의 파일이 없거나, 있어도 거의 비어 있음
- v5 자산이 새 구조에서도 경로/의미 충돌 없이 바로 사용 가능함
- 새 시스템의 정책/템플릿 기본값으로 즉시 사용 가능함

### 2. 병합
다음 조건을 만족하면 **병합** 대상으로 분류하라.
- `gdi_auto_work`에 이미 같은 목적의 파일이 존재함
- v5의 내용이 더 구체적이거나 품질이 높음
- 단순 덮어쓰기보다, 기존 TO-BE 구조와 결합하는 편이 맞음

### 3. 재작성
다음 조건을 만족하면 **재작성** 대상으로 분류하라.
- 경로 체계가 다름
- 상태 전이/명칭 체계가 다름
- v5 자산 내부에 현재 구조와 상충되는 규칙이 있음
- 의미는 재사용 가능하지만 파일 자체는 그대로 쓰기 어려움

### 4. 보류
다음 조건을 만족하면 **보류** 대상으로 분류하라.
- 현재 단계에서 직접 필요하지 않음
- analyze/build/fix/clean 핵심 흐름과 거리가 있음
- 지금 옮기면 구조 복잡도만 높아짐

### 1. dual-spec vs single-spec
문제:
- v5 일부 문서는 dual-spec을 기준으로 설명하지만, 일부 정책은 단일 spec 흐름처럼 보일 수 있다.
처리:
- `gdi_auto_work`의 canonical model은 dual-spec으로 고정한다.
- 사람 검토용 `people_spec.md`, 기계 입력용 `machine_spec.yml`을 중심으로 모든 자산을 맞춘다.

### 2. popup 파라미터 규칙 충돌
문제:
- v5 내부에는 popup 파라미터 수신 규칙이 일관되지 않을 수 있다.
처리:
- `gdi_auto_work`에서는 popup 파라미터 규칙을 한 가지 표준으로 통일한다.
- 충돌 규칙은 그대로 이식하지 말고 “재작성”으로 처리한다.

### 3. 로컬 절대경로 / 환경변수 / DB 접속정보
문제:
- v5 문서 및 settings에는 로컬 경로나 DB 접근 관련 환경값이 들어 있다.
처리:
- 절대경로, 민감정보, 로컬 개발환경 설정값은 복사 금지.
- 필요한 경우 `TODO: environment-specific`로만 남긴다.

## 실제 작업 순서

### Step 1. 대상 레포 구조 점검
- `gdi_auto_work` 현재 구조를 먼저 확인한다.
- 아래 경로가 실제 존재하는지 확인한다.
  - `system/policies/analyze/`
  - `system/policies/framework/`
  - `system/policies/runtime/`
  - `system/templates/spec/`
  - `system/templates/screen-types/`
  - `system/schemas/`
- 존재하지 않으면 디렉터리를 생성한다.

### Step 2. 복사 대상 선반영
먼저 아래 파일을 복사한다.
- `ppt_extraction.yml`
- `classify_tags.yml`
- `screen_spec_template.md` → `people_spec.md`
- `screen_spec_guide.md`

### Step 3. 병합 대상 반영
아래 파일은 기존 `gdi_auto_work` 파일과 비교 후 병합한다.
- `layout_rules.yml`
- `skeleton_contract.yml`
- `template_selection.yml`
- `screen_type_rules.yml`
- `screen-types/*.yml`

### Step 4. 재작성 대상 보정
아래 자산은 철학만 가져와 새 시스템 기준으로 재작성한다.
- `allowed_paths.yml`
- manifest schemas
- `.claude/commands/*`
- `.claude/skills/*`
- controller generation 관련 정책

### Step 5. 보류 대상 기록
- `/test`
- Playwright
- proposals 실제 내용 이식
위 항목은 `migration_report_v1.md`의 “보류 자산” 섹션에 기록만 남긴다.

### Step 6. 정합성 검증
반드시 아래를 검증하라.
- analyze 정책 파일이 실제 존재하는지
- `people_spec.md` 템플릿이 dual-spec 구조와 충돌하지 않는지
- screen_type 분류 결과명이 TO-BE 명칭 체계와 일치하는지
- template_selection과 screen-types 정의가 일치하는지
- skeleton_contract와 현재 skeleton placeholder 구조가 충돌하지 않는지
- runtime path 정책에 `archive/**` 실행 입력 금지가 유지되는지

   - 병합 필요 추가 검토: [파일 목록]

## 최종 완료 기준
1. v5의 analyze 핵심 자산이 `gdi_auto_work/system/policies/analyze/` 아래로 반영되어 있어야 한다.
2. `people_spec.md` 템플릿이 v5의 spec 템플릿 자산을 반영해 품질이 올라가 있어야 한다.
3. framework 정책 중 layout, template selection, skeleton contract, screen type 자산이 `gdi_auto_work` 구조에 맞게 병합되어 있어야 한다.
4. runtime/manifest/command 계열 자산은 경로 체계 충돌 없이 재작성 또는 보정 계획이 정리되어 있어야 한다.
5. `/test`, Playwright, proposals 등 후순위 자산은 보류 목록으로 명확히 분리되어 있어야 한다.
6. `migration_report_v1.md`와 `migration_todo_v1.md`가 작성되어 있어야 한다.
