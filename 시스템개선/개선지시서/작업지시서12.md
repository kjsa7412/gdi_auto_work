# v5 잔존 정책/규칙 자산 이관 및 시스템 정착 작업지시서

## 역할
당신은 `gdi_auto_work` 레포를 유지보수하는 수석 아키텍트이자 마이그레이션 실행 담당자다.  
이번 작업의 목적은 `C:\auto_gdi_work_v5`의 command/skill 본문에 아직 남아 있는 **정책/규칙 자산**을 `gdi_auto_work`의 현재 구조에 맞게 외부 정책 파일로 승격시키고, 반영된 정책이 **build / generate / fix / system-fix / schema / path / 산출물 구조 전체에 자연스럽게 안착**하도록 관련 부분까지 함께 정비하는 것이다.

---

## 작업 배경
이미 1차 이관으로 아래 항목은 반영된 상태다.

- `system/policies/analyze/ppt_extraction.yml`
- `system/policies/analyze/classify_tags.yml`
- `system/policies/analyze/screen_type_rules.yml`
- `system/templates/spec/people_spec.md`
- `system/policies/framework/layout_rules.yml`
- `system/policies/framework/skeleton_contract.yml` (여기에 일부 controller_generation 성격 병합됨)
- `system/policies/framework/template_selection.yml`
- `.claude/commands/*` 재작성
- `.claude/skills/*` 재작성

따라서 이번 작업은 **이미 반영된 항목을 다시 복사하는 작업이 아니다.**  
이번 작업은 v5의 command/skill 본문에 아직 하드코딩으로 남아 있는 정책/규칙을 찾아서, `gdi_auto_work`의 구조에 맞는 `system/policies/**`, `system/schemas/**`, `.claude/commands/**`, `.claude/skills/**`, 필요 시 `proposals/**` 등으로 **재배치/분리/연계**하는 작업이다.

---

## 최종 목표
다음 상태가 되도록 작업하라.

1. v5의 `generate.md`, `analyze.md`, `propose-policy-fix.md` 등에 남아 있는 정책/규칙이 command/skill 본문에서 빠져나와 **정책 파일** 또는 **관련 schema/skill 규칙**으로 분리되어 있을 것.
2. `gdi_auto_work`의 `generate` / `build` / `fix` / `system-fix` 가 새 정책 파일을 실제로 읽고 활용하는 구조가 될 것.
3. 새 정책이 생겼는데도 관련 schema, path, verify_result 구조, 오류처리 흐름이 구버전 상태로 남아 **반쪽 반영**되는 일이 없을 것.
4. 결과적으로 정책 파일과 command/skill의 역할이 분리되어, 이후 정책 추가/수정 시 command/skill을 매번 뜯어고치지 않아도 되게 만들 것.

---

## 절대 원칙

### 1. 복붙 이관 금지
v5 파일을 그대로 복사하지 마라.  
`gdi_auto_work`의 현재 구조와 철학에 맞춰 **의미를 이식**해야 한다.

### 2. command/skill는 흐름만 담당
정책/규칙의 상세 내용은 command/skill 본문에 두지 마라.  
가능한 한 `system/policies/**/*.yml` 또는 `system/schemas/**/*.json`에 외부화하라.

### 3. 정책만 만들고 끝내지 마라
새 정책 파일을 추가하면, 그 정책을 실제로 참조/검증/반환하는 관련 command/skill/schema/path/result 구조도 함께 수정하라.

### 4. 기존 1차 이관 결과를 깨지 마라
이미 반영된 `analyze / layout / template_selection / skeleton_contract / people_spec` 축을 훼손하지 마라.

### 5. 자유 생성 확대 금지
현재 `gdi_auto_work`의 skeleton 기반 생성 원칙을 유지하라.  
새 정책은 생성 자유도를 높이는 방향이 아니라 **생성 품질과 검증 강도**를 높이는 방향으로만 반영하라.

## 이번 작업의 핵심 이관 대상

### A. framework 계층으로 승격해야 할 정책
아래 정책은 현재 v5의 `generate.md` 본문에 남아 있으므로, `gdi_auto_work`의 framework 정책 파일로 분리하라.

#### A-1. SQL 생성 규칙
- 대상 파일 신규 생성: `system/policies/framework/sql_generation.yml`
- 담아야 할 내용:
  - 어떤 조건에서 어떤 SQL 산출물을 생성하는지
  - 메뉴등록 SQL / DDL 보완 SQL / 코드등록 SQL의 생성 조건
  - SQL 파일명 규칙
  - SQL 출력 위치 규칙
  - `execution_order.md`와의 연계 규칙이 필요하면 포함

#### A-2. 코드 등록 SQL 생성 규칙
- 대상 파일 신규 생성: `system/policies/framework/code_registration.yml`
- 담아야 할 내용:
  - `comm_cd`는 순수 코드값만 사용
  - 기존 `wrk_tp_cd + cd_tp_cd` 존재 여부 확인 후 신규 생성
  - 기존 코드그룹/타입 재사용 금지 조건
  - 신규 코드 제안과 실제 코드등록 SQL 생성의 구분 기준

#### A-3. 버튼 매핑 규칙
- 대상 파일 신규 생성: `system/policies/framework/button_mapping.yml`
- 담아야 할 내용:
  - 버튼 매핑 규칙
  - `listener.button.etc{N}.click` 규칙
  - `etc_desc{N}.click`류 자동 교정 규칙
  - 버튼 자동 매핑 실패 시 처리 기준

#### A-4. 코드 컴포넌트 기본값 규칙
- 대상 파일 신규 생성: `system/policies/framework/code_component_defaults.yml`
- 담아야 할 내용:
  - 코드콤보 / 코드헬프 컬럼 / DATA_VIEW / DATA_CODE_TYPE 기본값
  - `elements` 직접 선언 원칙
  - 코드명/코드값 검색 기준
  - 그리드 코드헬프 컬럼 기본 패턴

#### A-5. PostgreSQL 생성 규칙
- 대상 파일 신규 생성 권장: `system/policies/framework/postgresql_rules.yml`
- 단, 현재 구조상 더 자연스러우면 `system/policies/framework/xml_patterns.yml` 확장으로 처리 가능
- 담아야 할 내용:
  - 집계함수 내 윈도우함수 금지
  - 집계함수 중첩 금지
  - `STRING_AGG` + 서브쿼리 + DISTINCT 금지 패턴
  - 허용 대체 패턴
  - 자동 교정 가능한 경우와 수동 확인이 필요한 경우 구분

### B. verify 계층으로 승격해야 할 정책
현재 `gdi_auto_work`는 verify_result를 사용하지만, verify 정책 파일 계층이 비어 있거나 부족하다.  
v5 `generate.md`의 Integrated Verify 내용을 아래 파일로 외부화하라.

#### B-1. HTML 검증 규칙
- 신규 생성: `system/policies/verify/html_checks.yml`
- 포함 내용:
  - IIFE
  - `PGM` prefix
  - fragment include
  - 동적 id 규칙
  - 버튼 매핑
  - `etc{N}` listener 규칙
  - 코드콤보 / 코드헬프 컬럼 관련 검증
  - 그리드 이벤트 / popup 이벤트 / navigation 관련 핵심 검증

#### B-2. XML 검증 규칙
- 신규 생성: `system/policies/verify/xml_checks.yml`
- 포함 내용:
  - namespace 소문자
  - `java.util.Map`
  - audit 컬럼
  - `current_timestamp`
  - `_dts`
  - DDL 컬럼과 SQL 컬럼 일치
  - `COALESCE` 타입 통일
  - 금지 필터/금지 컬럼 규칙

#### B-3. SQL 검증 규칙
- 신규 생성: `system/policies/verify/sql_checks.yml`
- 포함 내용:
  - 메뉴등록 SQL 존재 여부
  - `sy_pgm_info` / `sy_menu_info` 관련 검증
  - `pgm_path` 규칙
  - 서브화면 등록 금지 기준
  - 감사컬럼 8개 확인
  - `execution_order.md` 존재 및 정합성 검증

#### B-4. Controller 검증 규칙
- 신규 생성: `system/policies/verify/controller_checks.yml`
- 포함 내용:
  - `@Slf4j @Controller`
  - 생성자 주입
  - `@AddUserInfo @LogAction @ResponseBody @RequestMapping`
  - `BaseResponse`
  - try-catch + `log.error`
  - CommonController 사용 여부 일관성

#### B-5. Service 검증 규칙
- 신규 생성: `system/policies/verify/service_checks.yml`
- 포함 내용:
  - `extends BaseService`
  - reader/writer 분리
  - writer 트랜잭션
  - namespace 규칙
  - CommonService 사용 여부 일관성

#### B-6. 자기진단 규칙
- 신규 생성: `system/policies/verify/self_diagnosis.yml`
- 포함 내용:
  - DDL 미존재 컬럼 사용 시 DDL보완 SQL 제안
  - 미등록 코드 발견 시 코드등록 SQL 제안
  - 불명확 항목 경고
  - 자동 수정 가능 / 불가능 구분

#### B-7. Verify 오류처리 / escalation 규칙
- 신규 생성: `system/policies/verify/escalation_rules.yml`
- 포함 내용:
  - critical 자동 수정 후 재검증
  - 자동 수정 불가 시 `manual_required: true`
  - warning 기록 기준
  - 반복 오류 발생 시 system-fix 또는 정책 제안 트리거 기준

#### C-1. Analyze 보완 우선순위

#### C-2. severity 기반 정책 반영 체계

## 관련 파일까지 함께 수정해야 하는 대응 범위
아래는 **반드시 같이 손대야 하는 연계 지점**이다.  
정책 파일만 만들고 아래를 수정하지 않으면 미완료로 간주한다.

### 1. `.claude/skills/generate.md`
다음을 수행하라.
- command/skill 본문에 남아 있는 정책 상세 문장을 제거 또는 축약하라.
- `system/policies/framework/*.yml`
- `system/policies/verify/*.yml`
- `system/config/naming.yml`
- `system/templates/**`
를 읽는 구조로 정리하라.
- Self-check를 단순 문자열 수준으로 두지 말고, **verify policy를 반영하는 구조**로 명시하라.
- 결과 반환 구조도 `verify_result`와 자연스럽게 연결되게 정리하라.

### 2. `.claude/commands/build.md`
다음을 확인/수정하라.
- generate 결과의 self-check / verify_result를 읽고 build 성공/실패 기준에 반영하라.
- `manual_required`, `critical_pass`, `failed checks` 등의 결과가 있으면 build 종료 정책이 명확해야 한다.
- 새 verify 정책 파일 계층을 build가 전제하도록 정리하라.

### 3. `.claude/commands/fix.md`
다음을 보강하라.
- 반복 오류, 구조적 오류, 정책 누락인 경우 `system-fix`를 강제 검토 또는 자동 호출하는 기준 추가
- 단순 산출물 patch와 정책 patch를 구분하는 분기 명확화
- verify / self_diagnosis / escalation 결과를 fix 판단에 활용하도록 반영

### 4. `.claude/skills/system-fix.md`
다음을 보강하라.
- severity 기반 자동 반영 / 제안만 반영 구분
- 정책 변경 backup
- 정책 제안 결과 저장 위치
- 정책/템플릿/skeleton/schema/runtime 중 어디를 수정할지 판정하는 기준
- 단발성 fix와 시스템적 fix의 경계 명확화

### 5. `system/schemas/**`
다음을 검토 및 수정하라.
- `verify_result.schema.json`
- 필요 시 `build_manifest.schema.json`, `fix_manifest.schema.json`, `machine_spec.schema.json`
- 새 verify 결과 구조가 schema에 반영되어야 한다.
- 특히 아래 필드 반영 검토:
  - `manual_required`
  - `critical_pass`
  - `auto_fixed`
  - `details.html_checks`
  - `details.xml_checks`
  - `details.sql_checks`
  - `details.controller_checks`
  - `details.service_checks`
  - `self_diagnosis`

### 6. `system/config/paths.yml` 및 runtime 관련 설정
다음을 점검하라.
- 새로 추가되는 `system/policies/verify/` 경로가 경로 체계에 자연스럽게 녹아 있는지 확인
- `proposals/policy_changes/`를 추가한다면 path/rule/allowed path에 반영
- runtime/allowed_paths와 충돌하지 않도록 정리

### 7. `system/policies/framework/skeleton_contract.yml` 및 screen-types/template 계열
다음을 점검하라.
- 새 SQL 생성 규칙이 현재 skeleton contract, artifact matrix, placeholder completeness 규칙과 충돌하지 않는지 확인
- SQL 산출물 구조를 skeleton_contract 또는 관련 matrix에 추가 보완할 필요가 있으면 반영
- code_component / button_mapping / verify 규칙 추가로 인해 screen-type 정의가 수정돼야 하는지 검토

### 8. README 또는 운영 문서
가능하면 최소한으로라도 반영하라.
- 정책 계층이 `analyze / framework / runtime`에서 `analyze / framework / runtime / verify`로 확장되었음을 문서화
- build / fix / system-fix의 정책 반영 흐름을 짧게라도 정리

## 완료 조건
아래를 모두 만족해야 완료다.

- v5의 잔존 정책이 command/skill 본문이 아니라 정책/스키마/연계 규칙으로 정리됨
- 새 정책 파일을 실제로 build/generate/fix/system-fix가 참조함
- verify 계층이 실질적으로 동작 가능한 구조가 됨
- verify_result 구조가 새 정책과 맞게 정리됨
- 정책만 추가되고 관련 path/schema/command가 구버전 상태로 남는 일이 없음
- 기존 1차 이관 결과와 충돌하지 않음

---

## 금지 사항
- 이미 완료된 1차 이관 항목을 다시 덮어쓰기 식으로 복사하지 마라.
- v5의 path 구조(`user/...`)를 현재 시스템에 그대로 되살리지 마라.
- command/skill 본문에 정책 원문을 다시 장문으로 집어넣지 마라.
- 새 정책 파일만 만들고 실제 참조처를 수정하지 않은 채 종료하지 마라.
- verify 정책을 만들고도 verify_result/schema/build 종료 조건을 방치하지 마라.

---

## 실행 시 판단 기준
판단이 애매하면 아래 기준을 따르라.

1. **정책인가?** → `system/policies/**/*.yml`
2. **구조 검증 계약인가?** → `system/schemas/**/*.json`
3. **실행 흐름/호출 순서인가?** → `.claude/commands/*.md`
4. **한 단계 내부 절차인가?** → `.claude/skills/*.md`
5. **반복 오류에 대한 시스템 보완 제안인가?** → `system-fix` 및 `proposals/policy_changes/`

---

## 최종 지시
위 요구사항을 누락 없이 수행하라.  
중요한 것은 **정책 파일 생성 자체가 아니라, 그 정책이 시스템 전체에 무리 없이 안착하도록 관련 command/skill/schema/path/result 구조까지 함께 반영하는 것**이다.  
부분 반영, 반쪽 이관, 문서만 있고 동작하지 않는 상태는 허용하지 않는다.
