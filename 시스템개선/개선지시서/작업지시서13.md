# 시스템 리팩토링 작업지시 프롬프트

너는 현재 `gdi_auto_work` 레포를 대상으로 **정책 중심 시스템 리팩토링**을 수행하는 수석 아키텍트이자 리팩토링 책임자이다.

이번 작업의 본질은 단순 정리나 파일 이동이 아니다.  
**command/skill 내부에 흩어져 있는 규칙, 제약, 강제사항, 판정 기준을 정책 자산으로 승격하여 `system/policies/**/*.yml` 중심 구조로 재편하는 시스템 리팩토링**이다.

---

## 작업의 목표

현재 시스템은 command와 skill이 절차와 규칙을 동시에 품고 있어, 규칙의 중복·누락·불일치가 발생할 수 있는 구조이다.  
이 상태를 개선하여 다음과 같은 구조로 재정렬하라.

- **정책(Policy)** = 시스템이 반드시 준수해야 하는 선언형 규칙의 SSOT
- **Command** = phase를 통제하는 얇은 오케스트레이터
- **Skill** = policy를 로드하여 실제 실행을 수행하는 executor
- **Manifest / Verify** = 어떤 policy를 어떤 근거로 준수했는지 추적하는 증적 계층

즉, 앞으로 시스템의 판단 기준은 문장형 md가 아니라 **정형화된 yml 정책**이 되어야 한다.  
command와 skill은 정책을 설명하는 문서가 아니라, 정책을 읽고 집행하는 얇은 실행 계층으로 축소되어야 한다.

---

## 핵심 리팩토링 원칙

1. **규칙은 정책으로 승격한다**
   - 반복적으로 등장하는 규칙
   - 생성 금지/허용 규칙
   - 입력 해석 규칙
   - 화면 유형 판정 규칙
   - skeleton 선택 규칙
   - placeholder 필수 계약
   - SQL 생성 규칙
   - 경로/수명주기/컨텍스트 격리 규칙
   - 검증 및 실패 판정 기준

   위 성격의 내용은 command/skill 본문에 남기지 말고 `system/policies/**/*.yml`로 승격한다.

2. **절차와 규칙을 분리한다**
   - command에는 실행 순서, phase 전환, 중단 조건, 호출 관계만 남긴다.
   - skill에는 입력, 실행 알고리즘, 반환 구조, 실패 처리만 남긴다.
   - 규칙의 상세 본문을 command/skill에 중복 기재하지 않는다.
   - “무엇을 지켜야 하는가”는 policy가, “언제 어떻게 수행하는가”는 command/skill이 담당한다.

3. **정책을 참조형 구조로 통일한다**
   - command/skill 내부에 규칙을 길게 서술하지 말고, 어떤 policy를 필수 로드해야 하는지 명시하는 방식으로 전환한다.
   - 모든 phase는 “반드시 준수해야 하는 policy 목록”을 갖도록 재구성한다.
   - command/skill은 정책 설명서가 아니라 정책 소비자여야 한다.

4. **정책은 선언형이어야 한다**
   - 정책은 사람이 읽을 수 있어야 하지만, 동시에 시스템이 강제할 수 있어야 한다.
   - 모호한 권고문이 아니라 id, 조건, action, severity, note, scope 등을 가진 정형 규칙 집합이어야 한다.
   - 예외가 있더라도 “예외 허용 조건”이 정책 안에서 드러나야 한다.

5. **정책 구조는 확장 가능해야 한다**
   - 특정 규칙 하나만 해결하는 임시 yml이 아니라, 이후 유사 규칙을 계속 수용할 수 있는 정책 체계여야 한다.
   - analyze / review / generate / sql / runtime / verify 등으로 정책 도메인을 분리할 수 있으면 분리한다.
   - 정책 파일명은 역할이 드러나야 하며, command/skill이 파일명을 보고 의미를 유추할 수 있어야 한다.

6. **정책 위반은 추적 가능해야 한다**
   - 어떤 command/skill이 어떤 policy를 읽었는지,
   - 어떤 rule id를 통과/실패했는지,
   - 실패 근거가 무엇인지
   가 manifest 또는 verify 산출물에 남도록 재구성한다.

---

## 리팩토링의 방향성

이번 작업은 “기존 md를 yml로 기계적으로 치환”하는 작업이 아니다.  
다음 관점으로 시스템을 재설계하라.

- command/skill에 남아 있는 규칙성 자산을 식별한다.
- 그 중 정책으로 승격되어야 하는 자산을 선별한다.
- 중복/충돌/모호/분산된 규칙을 통합한다.
- policy를 중심으로 command/skill의 책임을 다시 자른다.
- 앞으로 새로운 규칙이 생겨도 command 본문을 계속 비대화시키지 않고, policy 추가만으로 확장되도록 만든다.

특히 아래와 같은 유형은 반드시 정책 승격 대상으로 간주하라.

- `analyze` 계열: PPT 추출 규칙, 화면 판정 규칙, unresolved 처리 규칙
- `generate` 계열: skeleton 선택 규칙, placeholder 계약, artifact별 생성 제약
- `sql` 계열: 코드 등록 SQL 규칙, DDL 변경 대응 규칙, mapper 작성 규칙
- `runtime` 계열: allowed path, active context, lifecycle, db access, archive 배제
- `verify` 계열: 실패 기준, severity 기준, warn/error/fatal 판정 체계

---

## 기대하는 최종 상태

리팩토링 이후 시스템은 아래 상태를 만족해야 한다.

- command를 읽으면 절차와 phase 흐름이 보인다.
- skill을 읽으면 실행 책임과 입력/출력이 보인다.
- policy를 읽으면 시스템 규칙의 실체가 보인다.
- 같은 규칙이 command/skill 여러 곳에 중복 서술되지 않는다.
- 새 규칙 추가 시 command를 고치기보다 policy를 추가/수정하는 방식이 우선된다.
- 시스템 규칙의 기준점이 md 문장 묶음이 아니라 yml 정책 집합으로 고정된다.
- 사람이 “이 시스템은 무엇을 반드시 지키는가?”를 정책 폴더만 보고 파악할 수 있어야 한다.

---

## 작업 수행 시 절대 원칙

- 기존 시스템 철학을 깨지 말고, **기존 구조를 더 강한 정책 중심 구조로 정제**하라.
- command/skill을 제거하는 것이 목적이 아니다. **얇게 만들고 책임을 선명하게 만드는 것**이 목적이다.
- 규칙을 숨기지 말고, 정책 자산으로 드러내라.
- 임시방편 금지. 한 파일에서만 맞는 국소 수정 금지.
- 특정 요구사항만 맞추는 패치가 아니라, **향후 반복 수정에 견디는 구조적 개편**으로 처리하라.
- 문서 표현 정리가 아니라 **시스템 통제면(control plane) 리팩토링**이라는 관점으로 접근하라.

---

## 산출물의 성격

산출물은 단순 요약문이 아니라,  
**어떤 규칙이 어떤 policy로 승격되어야 하는지, 무엇은 command/skill에 남고 무엇은 policy로 이동해야 하는지, 시스템 책임이 어떻게 재편되어야 하는지**가 드러나는 리팩토링 결과여야 한다.

즉, 이번 작업은 기능 추가가 아니라  
**“규칙을 코드 밖에서 강제 가능한 정책 자산으로 재정렬하는 아키텍처 리팩토링”**이다.

이 기준에서 부족하거나 타협적인 결과는 허용하지 않는다.
