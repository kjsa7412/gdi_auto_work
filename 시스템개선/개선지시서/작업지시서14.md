# 시스템 정합성 리팩토링 작업지시서

너는 현재 `gdi_auto_work` 레포를 대상으로 **시스템 정합성 회복 및 구조 동기화 리팩토링**을 수행하는 수석 아키텍트이자 정비 책임자이다.

이번 작업은 단순 문서 수정이 아니다.  
또한 특정 파일 몇 개를 맞추는 국소 패치도 아니다.  
**문서, 실디렉토리, command, skill, manifest, policy, schema, template, runtime 상태 관리 방식 전반의 싱크를 재정렬하는 시스템 리팩토링**으로 수행하라.

---

## 작업의 본질

현재 시스템은 다음과 같은 위험을 내포할 수 있다.

- 문서에 설명된 구조와 실제 디렉토리 구조가 다를 수 있다.
- command/skill이 전제하는 파일, 경로, 정책, 산출물과 실제 저장소 구성이 다를 수 있다.
- manifest의 책임과 실제 생성/검증/배포 흐름이 어긋날 수 있다.
- README, 정책 문서, 명령 문서, 실행 흔적이 서로 다른 시스템상을 설명할 수 있다.
- 과거 설계 흔적과 현재 구현이 혼재되어 사용자가 무엇을 기준으로 믿어야 하는지 불명확할 수 있다.

이번 작업의 목적은 이러한 잠재 불일치를 개별적으로 땜질하는 것이 아니라,  
**시스템이 스스로 설명하는 구조와 실제로 동작하는 구조를 하나의 일관된 체계로 재정렬하는 것**이다.

---

## 최상위 목표

이번 작업은 아래 목표를 만족해야 한다.

1. **문서와 실제 구조의 기준점을 일치시킨다**
   - README, 가이드, command, skill, policy, manifest 설명이 실제 저장소 구조와 같은 시스템을 가리키도록 재정렬한다.

2. **시스템 책임 경계를 다시 선명하게 한다**
   - 어떤 파일이 구조를 설명하는지,
   - 어떤 파일이 규칙을 강제하는지,
   - 어떤 파일이 실행을 오케스트레이션하는지,
   - 어떤 파일이 실행 결과를 기록하는지,
   - 어떤 경로가 런타임 상태인지,
   - 어떤 경로가 영구 자산인지
   를 명확히 구분한다.

3. **실행 흐름과 기록 체계를 정합화한다**
   - analyze, review, build, fix, verify, deploy 준비 흐름과 manifest/summary/report 체계가 동일한 lifecycle을 설명하도록 만든다.

4. **중복 설명과 다중 진실원천을 제거한다**
   - 같은 규칙 또는 구조가 여러 문서에서 미세하게 다르게 설명되는 상태를 제거한다.
   - 시스템의 진실원천(SSOT)이 어디인지 명확히 고정한다.

5. **미래 변경에도 쉽게 깨지지 않는 구조로 만든다**
   - 앞으로 새 policy, 새 command, 새 manifest가 생겨도 문서와 구조가 다시 쉽게 어긋나지 않도록 정비한다.

---

## 리팩토링 관점

이번 작업은 다음 관점에서 수행하라.

### 1. 구조 정합성 관점
- 레포가 스스로 설명하는 구조와 실제 디렉토리/파일 구조가 하나의 모델로 수렴해야 한다.
- 문서에만 존재하는 개념, 실디렉토리에만 존재하는 개념, command 내부에서만 전제하는 개념을 분리해보지 말고 **하나의 시스템 모델로 통합**하라.

### 2. 실행 정합성 관점
- command가 설명하는 phase와 실제 산출물 구조가 같은 흐름을 말해야 한다.
- skill이 기대하는 입력과 manifest가 기록하는 출력이 같은 계약을 따라야 한다.
- runtime 정책, allowed path, active context, lifecycle 규칙이 실제 폴더 운영 방식과 충돌하지 않도록 정렬하라.

### 3. 정책 정합성 관점
- 규칙은 policy가 설명하고,
- command는 phase를 오케스트레이션하고,
- skill은 policy를 읽어 실행하고,
- manifest는 결과를 기록하는 구조로 정렬하라.
- 규칙이 md 본문, policy, manifest 설명, README에 제각각 흩어져 있지 않도록 하라.

### 4. 기록 정합성 관점
- manifest, verify result, summary, report가 서로 다른 기준으로 작성되지 않도록 하라.
- 기록 산출물은 “무엇이 수행되었는가”, “무엇이 생성되었는가”, “무엇을 근거로 판단했는가”를 일관되게 보여줘야 한다.

### 5. 운영 정합성 관점
- active/work/archive/cache/result/provided 등의 경로가 개념적으로 충돌하지 않아야 한다.
- 기록용 보관소와 시스템이 참조하는 실사용 경로를 명확히 구분하라.
- 런타임 상태가 문서상 설명과 실제 운영 방식에서 다르게 취급되지 않도록 하라.

## 반드시 지켜야 할 원칙

1. **실제 구조를 기준 없이 문서에 맞추는 식으로 왜곡하지 말 것**
   - 반대로 문서를 무시하고 현재 디렉토리만 정답으로 간주하지도 말 것
   - 시스템 철학, 실행 흐름, 유지보수성까지 고려하여 기준 구조를 재정의하라.

2. **국소 수정 금지**
   - README만 고치고 끝내지 말 것
   - command만 고치고 끝내지 말 것
   - manifest 명칭만 고치고 끝내지 말 것
   - 반드시 시스템 전반의 책임과 설명 체계를 함께 정리하라.

3. **중복된 설명을 남기지 말 것**
   - 동일한 구조 설명이 여러 파일에 존재한다면 역할을 재정의하고 중복을 제거하라.
   - 어떤 설명은 정책으로, 어떤 설명은 command로, 어떤 설명은 README로 올라가야 하는지 구분하라.

4. **설계와 실행이 분리되지 않도록 할 것**
   - 문서는 이상적인 미래 구조만 설명해서는 안 된다.
   - 실제 command/skill/manifest가 따라갈 수 있는 현실 구조여야 한다.

5. **이름, 경로, 책임의 일관성을 확보할 것**
   - 동일 개념이 문서마다 다른 이름으로 불리지 않도록 하라.
   - 동일 책임이 여러 경로에 나뉘어 중복 저장되지 않도록 하라.

6. **향후 확장성을 해치지 말 것**
   - 이번 정비 결과가 이후 policy 추가, command 추가, phase 추가에 견딜 수 있어야 한다.
   - 임시 예외 처리나 하드코딩성 구조 정리는 지양하라.

---

## 이 작업에서 재정렬되어야 하는 대상 범주

이번 작업은 최소한 아래 범주를 하나의 구조로 재정렬하는 관점에서 수행하라.

- 최상위 README 및 시스템 설명 문서
- 실디렉토리 구조 및 폴더 책임 정의
- `.claude/commands/*`
- `.claude/skills/*`
- `system/policies/**/*`
- `system/schemas/**/*`
- `system/templates/**/*`
- cache / runtime / work / archive 관련 경로 개념
- manifest / verify / summary / deploy prepare 산출물 체계
- active context / lock / lifecycle / allowed path 규칙

핵심은 개별 파일 수정이 아니라,  
**이들 전체가 하나의 동일한 시스템을 설명하고 실행하도록 책임과 기준을 재정비하는 것**이다.

---

## 기대하는 결과 상태

리팩토링 이후 시스템은 아래 상태여야 한다.

- README를 읽으면 현재 시스템 구조를 신뢰할 수 있다.
- 실디렉토리를 보면 문서와 동일한 책임 구성이 드러난다.
- command를 읽으면 phase 절차가 보인다.
- skill을 읽으면 실행 책임이 보인다.
- policy를 읽으면 시스템 규칙이 보인다.
- manifest를 보면 실제 수행 결과와 판정 근거가 보인다.
- runtime 경로와 archive 경로는 역할이 명확히 구분된다.
- 더 이상 “문서상 시스템”과 “실제 동작 시스템”이 따로 존재하지 않는다.

즉, 사용자는 어느 한 파일만 봐도 전체 구조를 오해하지 않아야 하며,  
시스템 유지보수자는 어떤 변경을 어디에 반영해야 하는지 즉시 판단할 수 있어야 한다.

---

## 수행 태도

- 이번 작업은 **싱크 회복 리팩토링**이다.
- 증상 수정이 아니라 구조 정렬로 접근하라.
- 문서 정리 수준으로 축소하지 말고, 시스템 통제면과 실행면이 같은 구조를 보도록 재편하라.
- “설명되는 시스템”, “실행되는 시스템”, “기록되는 시스템”을 하나로 맞추는 것이 핵심이다.

---

## 최종 지시

현재 `gdi_auto_work`를 대상으로,  
문서, 실디렉토리, command, skill, policy, schema, template, manifest, runtime 운영 방식 사이의 구조적 불일치를 예방하고 제거할 수 있는 방향으로 **시스템 정합성 리팩토링**을 수행하라.

이번 작업은 부분 수정이 아니라,  
**저장소가 스스로 설명하는 구조와 실제로 동작하는 구조를 일치시키는 아키텍처 정비 작업**으로 수행되어야 한다.

타협하지 말고, 미래 유지보수성과 정책 중심 운영까지 고려한 구조적 결과물을 만들어라.
