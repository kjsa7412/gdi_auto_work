작업 목적

이번 작업은 “.claude command 및 skill 구현 단계”이다.

이전 단계에서 작성된 framework 분석 결과, skeleton 자산, manifest/schema/runtime 계약을 바탕으로, 실제 Claude Code가 사용할 수 있는 /analyze, /build, /fix, /clean command 프롬프트와 generate, deploy, system-fix skill 프롬프트를 작성하라.

이번 단계의 목표는 “명령 실행 절차를 프롬프트 수준에서 완성”하는 것이다.

즉, 명령어가 어떤 파일을 읽고 어떤 정책을 적용하며 어떤 manifest를 갱신하고 어떤 산출물을 만들어야 하는지 Claude Code가 그대로 수행할 수 있도록 .md 파일로 구체화하라.

절대 규칙

1. command 는 반드시 이전 단계에서 정의한 manifest/schema/runtime 계약을 따르도록 작성하라.
2. command 는 자유로운 작업 지시 문장이 아니라, Claude Code가 따라야 할 명확한 절차형 프롬프트여야 한다.
3. command 내부에서 읽을 파일, 검증할 파일, 갱신할 manifest, 생성할 산출물, 실패 시 조치를 구체적으로 적어라.
4. single-active workspace 전제를 깨지 말아라.
5. archive 는 history only 영역이며 어떤 command 도 archive 를 실행 입력으로 사용하지 못하게 명시하라.
6. skeleton 없는 직접 생성은 금지하라.
7. framework policy 위반, schema 불일치, 필수 입력 누락, lock 충돌이 있으면 즉시 실패 처리하도록 명시하라.
8. analyze/build/fix/clean 각각의 책임을 섞지 말아라.
9. command 는 skill 을 호출할 수 있지만, skill 과 command 의 책임을 명확히 분리하라.
10. build 단계에서만 generate skill 을 사용하도록 하고, fix 는 필요 시 build 재실행 또는 system-fix skill 연계를 통해 해결하라.
11. deploy skill 은 환경이 불명확하면 “배포 준비/반영 대상 정리” 수준으로 설계하고, 외부 배포를 무조건 수행한다고 가정하지 말아라.
12. 보고서용 문구보다 실제 실행 지시가 우선이다.
13. command md 와 skill md 는 사람 설명서가 아니라 Claude Code 실행 프롬프트여야 한다.
14. 이전 단계의 파일명, 경로명, manifest 구조와 충돌하지 않게 작성하라.
15. 아직 일반-purpose 자율 수정 시스템으로 확장하지 말고, 현재 정의된 task/fix 흐름에 맞춰 최소 충분 범위로 작성하라.

작업 순서

1단계. 기존 계약 및 정책 재검토

이전 단계 산출물을 먼저 읽고, command 와 skill 작성에 필요한 실행 계약을 재정리하라.

특히 아래 파일을 우선 검토하라.

- system/config/paths.yml
- system/config/runtime.yml
- system/config/framework_manifest.yml
- system/config/naming.yml
- system/policies/runtime/*.yml
- system/policies/framework/*.yml
- system/templates/spec/people_spec.md
- system/templates/spec/machine_spec.yml
- system/schemas/*.json
- work/.active_context.yml
- work/task/2.Working/manifests/*.yml
- work/fix/2.Working/manifests/*.yml
- framework_analysis_report.md
- skeleton_extraction_report.md
- manifest_contract_report.md

검토 목적

- command 별 입력/출력/실패조건을 정확히 다시 정리
- skill 에 맡길 책임과 command 에서 직접 처리할 책임을 분리
- analyze/build/fix/clean 흐름이 manifest/state 와 자연스럽게 연결되는지 확인
- lock, active context, allowed_paths, archive 금지 규칙을 프롬프트에 어떻게 녹일지 확정

이 단계에서 부족한 항목은 “command 구현 보완 목록”으로 정리한 뒤 이후 단계에서 반영하라.

2단계. command 와 skill 책임 분리 정의

아래 구분을 기준으로 책임을 명확히 정리하라.

command 책임

- 진입 조건 확인
- active context 및 lock 확인/갱신
- 필수 입력 존재 확인
- schema/policy 기반 선행 검증
- 수행 절차 orchestration
- 필요한 skill 호출
- manifest/status 갱신
- 실패 시 중단 및 기록
- 결과 보고 요약 작성

skill 책임

- 특정 하위 작업의 세부 실행
- generate: skeleton 선택 및 placeholder 치환, artifact 생성
- deploy: 산출물 정리 및 반영 대상 패키징 또는 배포 준비
- system-fix: 반복 오류의 원인 분류 및 정책/템플릿/계약 보완 제안

중요:

command 는 orchestration 중심으로 작성하고, 실질적 생성/반영 로직은 skill 로 분리하라.

단, analyze 는 별도 skill 없이 command 내부 절차로 수행해도 된다면 그렇게 설계해도 되지만, build 는 반드시 generate skill 중심으로 설계하라.

3단계. /analyze command 작성

.claude/commands/analyze.md 를 작성하라.

반드시 포함할 것

- 명령 목적
- 실행 전 확인 사항
- lock 확인 규칙
- active context 확인 및 상태 갱신 규칙
- 입력 경로: work/task/1.Prep
- 허용 입력 확장자: ppt/pptx
- 금지 입력: archive, work/fix, 외부 미정의 경로
- 필수 참조 파일
    - system/config/*.yml
    - system/policies/analyze 가 있다면 그것
    - system/policies/runtime/*.yml
    - system/templates/spec/*
    - system/cache/convention/*
    - system/templates/screen-types/*
    - framework 관련 정책 파일
- analyze 수행 절차
    1. workspace 및 lock 검증
    2. task prep 입력 확인
    3. active_context 상태 변경
    4. analyze.manifest 초기화 또는 갱신
    5. ppt 추출 결과 정리
    6. 화면 후보 식별
    7. screen_type 분류
    8. people_spec original 생성
    9. machine_spec original 생성
    10. final people_spec 초기 복사본 생성
    11. unresolved 기록
    12. analyze.manifest 완료 상태 갱신
    13. review 대상 경로 안내
- 실패 조건
    - 입력 없음
    - lock 충돌
    - screen_type 분류 불가
    - schema 불일치
    - 필수 spec 생성 실패
- 성공 조건
    - people_spec original/final 생성
    - machine_spec original 생성
    - analyze.manifest 정상 갱신
    - active_context 상태 task_analyzed 반영

중요:

analyze command 는 아직 코드를 생성하지 않는다.

사용자가 검토해야 할 파일은 final people_spec 임을 명확히 적어라.

4단계. /build command 작성

.claude/commands/build.md 를 작성하라.

반드시 포함할 것

- 명령 목적
- task_reviewed 상태 또는 이에 준하는 실행 가능 조건
- 필수 입력
    - work/task/2.Working/original/people_spec.md
    - work/task/2.Working/original/machine_spec.yml
    - work/task/2.Working/final/people_spec.md
- 금지 입력
    - archive
    - 임의 직접 작성 코드
    - skeleton 미정의 artifact
- 필수 참조 파일
    - framework policies
    - template_selection
    - skeleton_contract
    - screen-types
    - skeleton files
    - runtime policies
    - schemas
- build 수행 절차
    1. lock 및 active_context 검증
    2. build.manifest 초기화 또는 갱신
    3. original/final people_spec diff 분석
    4. diff 를 반영한 final machine_spec 생성
    5. machine_spec schema 검증
    6. generate skill 호출
    7. verify_result 생성
    8. 위반 시 즉시 보정 시도 또는 실패 처리
    9. deliverables/report 정리
    10. build.manifest 완료 상태 갱신
    11. active_context 상태 task_built 반영
- generate skill 호출 기준
    - screen_type 이 명확해야 함
    - skeleton selection 이 확정되어야 함
    - 필수 placeholder 가 모두 채워져야 함
- 실패 조건
    - final people_spec 누락
    - machine_spec 생성 실패
    - skeleton 없음
    - verify_result 에 error 존재
    - forbidden pattern 검출
- 성공 조건
    - final machine_spec 생성
    - generated artifacts 생성
    - verify_result 완료
    - deliverables/report 배치 완료

중요:

build command 는 자유 생성이 아니라 반드시 skeleton + template_selection + skeleton_contract 기반 생성임을 강하게 명시하라.

5단계. /fix command 작성

.claude/commands/fix.md 를 작성하라.

반드시 포함할 것

- 명령 목적
- 필수 입력: work/fix/1.Prep/fix.md
- 참조 가능 대상
    - work/task/2.Working/**
    - work/task/3.Result/**
    - system/policies/**
    - system/templates/**
- 금지 대상
    - archive
    - task prep 원본 덮어쓰기
    - 무근거 직접 patch
- fix 수행 절차
    1. lock 및 active_context 검증
    2. fix.manifest 초기화 또는 갱신
    3. fix.md 내용 분석
    4. 오류 유형 분류
        - 데이터 해석 오류
        - skeleton/생성 로직 오류
        - framework policy 누락
        - runtime 계약 누락
    5. 영향 범위 판정
        - reanalyze_required
        - rebuild_required
        - policy_patch_required
    6. 필요한 경우 system-fix skill 호출
    7. 필요한 경우 build 재유도 또는 analyze+build 재유도
    8. fix verify_result 생성
    9. fix.manifest 완료 상태 갱신
    10. active_context 상태 fix_applied 반영
- 실패 조건
    - fix 입력 없음
    - 관련 task 산출물 없음
    - 영향 범위 판정 불가
    - 재생성 실패
- 성공 조건
    - fix.manifest 작성
    - 영향 범위 명확화
    - 필요한 재실행 또는 보정 완료
    - 결과 및 보고서 생성

중요:

fix command 는 단순 텍스트 수정 명령이 아니라, 재발 방지 관점에서 system-fix skill 연계 가능성을 항상 검토하도록 작성하라.

6단계. /clean command 작성

.claude/commands/clean.md 를 작성하라.

반드시 포함할 것

- 명령 목적
- single-active workspace 정리 목적
- archive 는 history only 임을 재명시
- clean 수행 절차
    1. lock 및 current state 확인
    2. archive timestamp 생성
    3. work/task, work/fix 전체를 archive/work_YYYYMMDD_HHMMSS 로 이동
    4. directory shape 유지
    5. work/.active_context.yml 초기화
    6. work/.lock 초기화
    7. clean 결과 요약 기록
- 금지 사항
    - archive 내용 삭제
    - archive 를 재입력으로 읽기
    - 일부만 정리하고 일부 active 잔존
- 성공 조건
    - work 가 초기화됨
    - archive 보관 완료
    - active_context 가 idle 로 초기화됨
    - lock 이 UNLOCKED 로 복귀함

중요:

clean command 는 destructive 하므로, archive 보관을 먼저 수행하고 reset 하도록 강하게 적어라.

7단계. generate skill 작성

.claude/skills/generate.md 를 작성하라.

반드시 포함할 것

- skill 목적
- build command 로부터 받는 입력
    - final machine_spec
    - template_selection rules
    - skeleton_contract
    - screen-type 정의
    - skeleton files
    - framework policies
- generate 수행 절차

    4. skeleton_contract 기준으로 placeholder completeness 확인
    5. placeholder 치환
    6. html/xml/controller/service artifact 생성
    7. framework forbidden pattern/self-check 수행
    8. generated_paths 정리
    9. build.manifest/verify_result 에 반영할 정보 반환
- 실패 조건
    - 필수 placeholder 누락
    - skeleton 파일 없음
    - screen_type 불명확
    - framework policy 위반
- 성공 조건
    - artifact plan 에 맞는 파일 생성
    - generated_paths 목록 생성
    - self-check 결과 반환

중요:

generate skill 은 절대 자유 생성하지 말고 skeleton 파일을 직접 사용하도록 명시하라.

8단계. deploy skill 작성

.claude/skills/deploy.md 를 작성하라.

반드시 포함할 것

- skill 목적
- 이번 단계에서는 외부 환경이 확정되지 않았을 수 있으므로 deploy 를 “배포 준비 또는 반영 대상 정리” 중심으로 설계
- 입력
    - build 결과물
    - deliverables 경로
    - 필요한 경우 target path 정보
- deploy 수행 절차 예시
    1. deploy 대상 파일 확인
    2. artifact 존재 여부 검증
    3. 반영 대상 구조 정리
    4. overwrite 위험 경고
    5. 배포 준비 결과 또는 반영 결과 기록
- 출력
    - deploy summary
    - copied/prepared paths
    - warnings
- 실패 조건
    - deliverables 없음
    - target path 미확정
    - 충돌 위험 해결 불가

중요:

환경 근거 없이 무조건 실제 배포를 수행한다고 쓰지 말아라.

배포 환경이 아직 미확정이면 deploy skill 은 준비/패키징/반영대상 정리 수준으로 작성하라.

9단계. system-fix skill 작성

.claude/skills/system-fix.md 를 작성하라.

반드시 포함할 것

- skill 목적
- 반복 오류 또는 구조적 오류를 시스템 차원에서 흡수
- 입력
    - fix.md 내용
    - fix.manifest
    - build/analyze verify_result
    - 관련 policy/template/spec 경로
- 수행 절차
    1. 오류 원인 분류
    2. 일회성 데이터 오류 vs 구조적 반복 오류 구분
    3. 보완 대상 식별
        - framework policy
        - template_selection
        - skeleton_contract
        - skeleton template
        - runtime rule
        - manifest contract
    4. 자동 패치 가능/불가 판단
    5. patch proposal 또는 직접 보정 지시 생성
    6. fix.manifest 에 policy_patch_required 반영할 정보 반환
- 출력
    - issue classification
    - recommended patch targets
    - auto-fix 가능 여부
    - follow-up actions
- 실패 조건
    - 원인 분류 불가
    - 관련 근거 파일 부족

중요:

system-fix skill 은 “프로그램 전체적으로 반영되어야 하는 수정사항”을 담당하는 방향으로 작성하라.

10단계. .claude/settings.json 작성

.claude/settings.json 을 작성하거나 보정하라.

반드시 반영할 것

- command 와 skill 경로
- single-active workspace 전제
- archive input 금지 원칙
- work 중심 실행 원칙
- skeleton 기반 생성 원칙
- runtime manifest/schema 검증 우선 원칙

실제 Claude Code 설정 형식이 프로젝트에서 이미 정해져 있지 않다면, 최소 설정 초안 형태로 작성하라.

미확정인 항목은 TODO 주석 또는 설명으로 남겨라.

11단계. command 와 skill 간 정합성 역검증

작성한 analyze/build/fix/clean command 와 generate/deploy/system-fix skill 이 서로 모순되지 않는지 역검증하라.

반드시 확인할 것

- build command 와 generate skill 의 입출력 연결
- fix command 와 system-fix skill 의 역할 구분
- clean command 와 runtime lifecycle 충돌 여부
- analyze/build/fix/clean 이 active_context 및 lock 규칙과 일치하는지
- command 내부 참조 파일 경로가 paths/runtime/policy/schema 와 일치하는지
- command 들이 archive input 금지 규칙을 모두 반영하는지
- build 이후 deploy skill 이 선택적으로 연결 가능한 구조인지

불일치가 있으면 command 또는 skill md 를 수정해서 맞춰라.

12단계. 최종 보고서 작성

command_implementation_report.md 파일을 작성하라.

반드시 아래 목차를 포함하라.

1. 작업 개요
2. 입력으로 사용한 이전 단계 결과물
3. command 구현 방침
4. /analyze 설계 요약
5. /build 설계 요약
6. /fix 설계 요약
7. /clean 설계 요약
8. generate skill 설계 요약
9. deploy skill 설계 요약
10. system-fix skill 설계 요약
11. settings.json 반영 내용
12. command-skill 정합성 검증 결과
13. 아직 미해결인 항목
14. 다음 단계에서 해야 할 작업

가능하면 어떤 파일의 어떤 정책을 어느 command/skill 이 참조하는지 경로 기준으로 명시하라.

출력 방식

1. 실제 파일 생성이 가능하면 지정된 경로에 파일을 생성하라.
2. 실제 파일 생성이 불가능하면 파일별 내용을 구분해서 전부 출력하라.
3. 보고서만 쓰고 끝내지 말고, 반드시 command md, skill md, settings.json 초안까지 함께 작성하라.
4. 미확정 항목은 “미확정” 또는 “근거 부족”으로 표시하라.
5. 단계 종료 시 작성 또는 수정한 파일 목록을 반드시 정리하라.

최종 완료 기준

1. /analyze, /build, /fix, /clean command md 가 작성되어 있어야 한다.
2. generate, deploy, system-fix skill md 가 작성되어 있어야 한다.
3. command 와 skill 이 manifest/schema/runtime/framework policy 와 연결되어 있어야 한다.
4. build 가 generate skill 을 통해 skeleton 기반 생성으로 이어지도록 정의되어 있어야 한다.
5. fix 가 system-fix skill 과 연계 가능한 구조로 정의되어 있어야 한다.
6. clean 이 archive-first 원칙으로 정의되어 있어야 한다.
7. .claude/settings.json 초안이 작성되어 있어야 한다.
8. command_implementation_report.md 에 정합성 검증 결과와 다음 단계 작업이 포함되어 있어야 한다.

작업 시작 전 선언

이번 작업은 “.claude command 및 skill 구현 단계”이며, 실제 업무 산출물 생성 실행 단계가 아니다.

따라서 실제 특정 task/fix 입력을 넣고 analyze/build/fix 를 수행하지 말고, 다음 단계에서 바로 사용할 수 있는 command 프롬프트, skill 프롬프트, settings 초안만 작성하라.
