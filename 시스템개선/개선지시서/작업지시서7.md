작업 목적

이번 작업은 system/cache/convention/ 을 단순 보관 디렉터리가 아니라, analyze/build/fix 실행 체인에서 실제로 갱신되고 참조되는 운영 자산으로 연결하는 수정 작업이다.
현재 구조에서는 system/cache/convention/ 경로와 cache-first 정책은 존재하지만, 어떤 단계가 언제 cache 를 갱신하고, 어떤 단계가 어떤 조건에서 cache 를 우선 참조하는지 command/skill/manifest 수준에서 연결이 약하다.
따라서 이번 작업에서는 convention cache 의 역할을 명확히 정의하고, analyze/build/fix 명령과 관련 policy/report/schema/manifest 샘플에 반영하여 실제 실행 체인에 편입시켜라.

중요 전제

1. 이번 작업은 cache 연결 강화 작업이다.
2. framework 분석 결과, skeleton 자산, manifest/schema/runtime 구조 자체를 뒤엎지 말고, 현재 구조를 유지한 상태에서 convention cache 를 실행 체인 안으로 편입시켜라.
3. cache 는 기본적으로 code/table/view/function/trigger 정의를 빠르게 참조하기 위한 로컬 자산이며, DB 직접 조회보다 우선한다.
4. DB 직접 조회는 cache-first 원칙 하에서 부족한 경우에만 허용한다.
5. archive 는 여전히 history only 이며 실행 입력으로 사용 금지다.
6. single-active workspace 전제를 유지한다.

현재 문제 인식

아래 문제가 존재한다고 가정하고 수정하라.

1. system/cache/convention/ 경로와 파일은 존재하지만, /analyze 본문에 cache refresh 절차가 구체적으로 없다.
2. /build 와 /fix 에서 convention cache 참조 조건이 약하거나 명시되지 않았다.
3. system/cache/convention/sql/*.sql 의 사용 시점과 산출물(code.txt, table.txt 등) 갱신 책임이 명확하지 않다.
4. analyze.manifest/build.manifest/fix.manifest 어디에도 cache refresh 여부, cache 사용 여부, DB fallback 여부가 충분히 남지 않는다.
5. report 문서에도 convention cache 의 역할과 실행 시점이 약하게 기술되어 있다.

핵심 목표

1. analyze 단계에서 convention cache 를 언제 갱신하는지 명확해져야 한다.
2. build 단계에서 machine_spec 보완 또는 naming/sql id/구조 검증이 필요할 때 convention cache 를 먼저 참조하도록 해야 한다.
3. fix 단계에서 오류 분석 시 convention cache 를 먼저 확인하고, 필요 시 DB fallback 여부를 판정하도록 해야 한다.
4. manifest 에 cache refresh 여부, cache 사용 여부, DB fallback 여부가 기록되어야 한다.
5. 정책 문서와 command 프롬프트, manifest 샘플, report 문서가 서로 모순되지 않아야 한다.

작업 순서

1단계. 기존 convention cache 관련 정의 전수 점검

아래 파일을 먼저 읽고, 현재 convention cache 와 관련된 정의가 어디까지 연결되어 있는지 점검하라.

- system/config/paths.yml
- system/config/runtime.yml
- system/policies/runtime/db_access.yml
- system/policies/runtime/allowed_paths.yml
- system/policies/runtime/context_isolation.yml
- .claude/commands/analyze.md
- .claude/commands/build.md
- .claude/commands/fix.md
- .claude/skills/system-fix.md
- system/cache/convention/sql/*.sql
- work/task/2.Working/manifests/analyze.manifest.yml
- work/task/2.Working/manifests/build.manifest.yml
- work/fix/2.Working/manifests/fix.manifest.yml
- manifest_contract_report.md
- command_implementation_report.md

이 단계에서 반드시 정리할 것

1. cache 를 현재 직접 읽는 command/skill 이 무엇인지
2. cache refresh 를 수행한다고 명시된 곳이 있는지
3. DB fallback 이 어느 단계에서 허용되는지
4. cache 사용 여부가 manifest 에 남는지
5. cache 관련 규칙이 정책/command/report 간 충돌하는지

이 단계 결과는 “convention cache 연결 누락 목록”으로 정리하라.

2단계. convention cache 역할 정의 명확화

system/cache/convention/ 의 역할을 실행 관점에서 명확히 정의하라.
반드시 아래를 정리하라.

1. code.txt 의 역할
2. table.txt 의 역할
3. view.txt 의 역할
4. function.txt 의 역할
5. trigger.txt 의 역할
6. sql/*.sql 의 역할
7. cache refresh 주체
8. cache refresh 시점
9. cache 사용 우선순위
10. cache 신뢰 범위
11. DB fallback 허용 조건

권장 원칙

- analyze 는 기본적으로 시작 초기에 cache refresh 필요 여부를 판정한다.
- build 는 cache refresh 를 기본 수행하지 않고, 기존 cache 를 참조한다.
- fix 는 기본적으로 cache 참조 후, 구조적 mismatch 가 있으면 system-fix 또는 DB fallback 여부를 판단한다.
- DB 직접 조회는 cache-first 후에만 가능하다.

이 정의를 system/config/runtime.yml 과 system/policies/runtime/db_access.yml 에 반영하라.

3단계. analyze 실행 체인에 cache refresh 연결

/analyze command 를 수정하여 convention cache refresh 를 실행 체인에 명시적으로 편입시켜라.

반드시 포함할 것

1. analyze 시작 시 cache refresh 필요 여부 판정 단계
2. refresh 조건 예시
   - cache 파일이 비어 있음
   - cache 파일이 누락됨
   - 최근 refresh 기록이 없음
   - 사용자가 refresh 를 요구함
   - framework/reference source 변경이 감지됨
3. refresh 수행 방식
   - system/cache/convention/sql/*.sql 을 사용하여 최신 정의를 조회한다.
   - 결과를 code.txt, table.txt, view.txt, function.txt, trigger.txt 에 반영한다.
4. refresh 실패 시 처리
   - cache 기존본이 있으면 warning 후 계속 진행할지
   - cache 가 전혀 없으면 analyze 실패로 처리할지
5. refresh 결과 기록
   - refresh 수행 여부
   - refresh 시각
   - refresh 대상 파일
   - refresh 실패/성공 상태
   - DB fallback 사용 여부

/analyze command 에서는 단순히 “cache 참조”라고만 적지 말고, 실제로 cache refresh 확인 → 필요 시 refresh → 이후 cache 참조 순으로 단계가 보여야 한다.

4단계. build 실행 체인에 convention cache 참조 연결

/build command 를 수정하여 convention cache 참조 조건을 명시하라.

반드시 포함할 것

1. final machine_spec 생성 또는 보정 시 naming/sql id/DDL/code 정의 확인이 필요하면 cache 를 먼저 참조한다.
2. cache 에서 해결 가능한 경우 DB 직접 조회를 하지 않는다.
3. build 는 기본적으로 refresh 를 하지 않는다. 단, cache 부재 또는 명백한 불일치 시에만 warning 또는 실패 처리한다.
4. build manifest 또는 verify_result 에 아래 정보가 남아야 한다.
   - cache used: true/false
   - cache categories used: code/table/view/function/trigger
   - cache insufficient: true/false
   - db fallback used: true/false
5. generate skill 에 넘기기 전에 cache 기반 검증이 필요한 포인트를 정리한다.

예시 포인트

- screen_id 또는 프로그램 규칙 검증
- mapper namespace 또는 sql id 확인
- 컬럼/테이블 존재성 검증
- 코드 정의 명칭 보완

5단계. fix 실행 체인에 convention cache 참조 연결

/fix command 와 system-fix skill 을 수정하여 convention cache 를 문제 분석 체인에 연결하라.

반드시 포함할 것

1. fix 분석 시 관련 오류가 naming/DDL/sql id/구조 mismatch 이면 cache 를 먼저 조회한다.
2. cache 로 원인 분류 가능한 경우 DB fallback 없이 진행한다.
3. cache 로도 판정이 불가능하면 db_access 정책에 따라 fallback 허용 여부를 판정한다.
4. fix.manifest 에 아래 정보가 남아야 한다.
   - cache used
   - cache categories used
   - cache mismatch detected
   - db fallback used
   - policy_patch_required 와의 관계
5. system-fix skill 은 cache 와 실제 산출물 간 불일치를 구조적 오류의 근거로 활용할 수 있어야 한다.

6단계. runtime policy 보정

아래 파일을 수정하여 convention cache 규칙이 명시적으로 드러나게 하라.

- system/config/runtime.yml
- system/policies/runtime/db_access.yml
- system/policies/runtime/allowed_paths.yml
- system/policies/runtime/context_isolation.yml

반드시 반영할 것

1. analyze/build/fix 각 단계의 cache 읽기 가능 경로
2. analyze 의 cache refresh 허용 범위
3. build/fix 의 기본 동작은 read-only cache 참조임을 명시
4. DB fallback 은 cache-first 이후만 허용
5. archive 는 cache refresh source 가 될 수 없음
6. cache refresh 결과 기록 위치
7. cache 관련 failure / warning 정책

필요 시 cache_refresh.yml 같은 별도 policy 를 추가해도 된다.

출력 방식

1. 실제 파일 생성이 가능하면 지정된 경로에 파일을 생성하라.

3. 보고서만 쓰고 끝내지 말고, command/policy/schema/manifest 샘플 수정본까지 함께 작성하라.
4. 미확정 항목은 “미확정” 또는 “근거 부족”으로 표시하라.
5. 단계 종료 시 작성 또는 수정한 파일 목록을 반드시 정리하라.

최종 완료 기준

1. /analyze 에 cache refresh 절차가 명시적으로 포함되어 있어야 한다.
2. /build 와 /fix 에 cache 참조 규칙이 명시되어 있어야 한다.
3. db_access 정책이 cache-first 원칙을 실행 절차 수준으로 설명해야 한다.
4. analyze/build/fix manifest 와 schema 에 cache 관련 필드가 추가되어 있어야 한다.
5. verify_result 에 cache 관련 체크/경고가 남을 수 있어야 한다.
6. report 문서에 convention cache 의 실행 체인 역할이 반영되어 있어야 한다.
7. cache_integration_report.md 가 작성되어 있어야 한다.

작업 시작 전 선언

이번 작업은 “system/cache/convention 실행 체인 통합 수정 단계”이다.
따라서 새로운 아키텍처를 다시 설계하지 말고, 기존 구조를 유지한 상태에서 convention cache 를 analyze/build/fix 흐름에 명시적으로 연결하는 수정만 수행하라.
