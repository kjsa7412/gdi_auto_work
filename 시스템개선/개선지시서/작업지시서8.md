작업 목적

이번 작업은 gdi_auto_work 레포에서 system/cache/convention/ 절차를 실제 실행 체인에 완전히 연결하고, cache 에 정보가 없거나 부족한 경우 실제 DB 조회를 통해 보완하는 절차를 명시적으로 추가하는 수정 작업이다.
현재 레포에는 cache-first 원칙과 DB lookup 허용 정책이 존재하지만, /analyze, /build, /fix command 본문에는 “cache 부재/부족 시 DB 접근 후 보완” 절차가 충분히 구체화되어 있지 않다.
따라서 이번 작업에서는 기존 구조를 유지한 상태에서, convention cache 와 DB fallback 절차를 command, policy, schema, manifest 샘플, report 문서에 일관되게 연결하라.

현재 확인된 상태

1. system/policies/runtime/db_access.yml 에는 다음 원칙이 이미 존재한다.
- default_mode: cache-first
- sync: analyze 에서 convention cache refresh 가능
- lookup: analyze/build/fix 에서 cache 부족 시 보완 가능
- 직접 DB lookup 전에 grep 또는 cache lookup 선행

2. /analyze command 는 system/cache/convention/** 를 참조 파일로 읽도록 적혀 있다.

3. 그러나 /analyze, /build, /fix 본문에는 아래가 충분히 구체적으로 내려와 있지 않다.
- cache refresh 필요 여부 판정
- cache 가 비어 있거나 정보가 없을 때 실제 DB 접근 수행 절차
- DB 접근 결과를 cache 에 어떻게 반영하는지
- manifest/verify_result 에 cache miss 와 DB fallback 이 어떻게 기록되는지

절대 규칙

1. 이번 작업은 cache 와 DB fallback 연결 강화 작업이다. 기존 아키텍처를 다시 설계하지 말아라.
2. 기본 원칙은 항상 cache-first 이다.
3. 실제 DB 접근은 cache 에 정보가 없거나 부족하거나 불일치가 명확한 경우에만 허용한다.
4. DB 접근은 select 계열 조회만 허용한다. insert/update/delete/ddl 은 금지한다.
5. archive 는 여전히 history only 이며 실행 입력으로 사용 금지다.
6. single-active workspace 전제를 유지한다.
7. command, policy, schema, manifest 샘플, report 문서가 서로 같은 규칙을 말해야 한다.
8. 새 규칙은 analyze/build/fix 의 실제 실행 절차 안에 자연스럽게 연결되어야 한다.

핵심 목표

1. analyze 단계에서 cache refresh 와 cache miss 시 DB 접근 절차가 명시되어야 한다.
2. build 단계에서 naming/sql id/DDL/코드 정의 확인 시 cache 우선 조회 후, cache 부족 시에만 DB fallback 하도록 해야 한다.
3. fix 단계에서 오류 원인 분석 시 cache 를 먼저 보고, cache 에 정보가 없거나 구조 mismatch 가 확인되면 DB fallback 을 판단해야 한다.
4. manifest 와 verify_result 에 cache hit/miss, DB fallback 여부, fallback 사유가 기록되어야 한다.

아래 파일을 먼저 읽고, cache-first 와 DB lookup 규칙이 실제로 어디까지 연결되어 있는지 점검하라.

반드시 정리할 것

1. cache 를 읽는 단계가 어디인지
2. cache refresh 를 누가 수행하는지
3. cache miss 시 DB fallback 이 어디서 허용되는지
4. manifest 에 cache 관련 기록이 남는지
5. command/policy/report 간 충돌이 있는지

이 단계 결과를 “cache 및 DB fallback 연결 누락 목록”으로 정리하라.

2단계. cache hit / miss / fallback 용어와 기준 정의

아래 용어를 정책 수준에서 명확히 정의하라.

1. cache hit
- 필요한 code/table/view/function/trigger 정보가 cache 파일에서 충분히 확인된 상태

2. cache miss
- 필요한 항목이 cache 파일에 전혀 존재하지 않는 상태

3. cache insufficient
- 일부는 존재하지만 build/fix/analyze 판단에 필요한 수준으로 충분하지 않은 상태

4. cache mismatch
- cache 정보와 현재 분석 대상 또는 생성 대상 구조가 충돌하는 상태

5. db fallback
- cache miss 또는 cache insufficient 또는 cache mismatch 를 해소하기 위해 실제 DB select 조회를 수행하는 절차

이 정의를 system/config/runtime.yml 또는 별도 runtime policy 에 반영하라.

3단계. analyze 체인에 “cache 없으면 DB 접근” 절차 추가

/analyze command 를 아래 기준으로 수정하라.

반드시 포함할 것

1. analyze 시작 직후 cache 상태 점검 단계
- code.txt, table.txt, view.txt, function.txt, trigger.txt 존재 여부 확인
- 파일이 비어 있는지 확인
- 필요한 최신성 기준이 있는지 확인

2. cache refresh 필요 여부 판정
예시 조건
- cache 파일 누락
- cache 파일 0 byte 또는 사실상 비어 있음
- 최근 refresh 시각 없음
- 사용자가 refresh 를 명시적으로 요구
- framework/reference source 변화로 refresh 필요성이 높음

3. cache refresh 절차
- 우선 system/cache/convention/sql/*.sql 기반으로 필요한 조회를 수행한다.
- 조회 결과를 해당 txt 파일에 반영한다.
- refresh 결과를 manifest 에 기록한다.

4. refresh 후에도 필요한 정보가 cache 에 없으면 DB fallback 수행
- 필요한 항목이 code/table/view/function/trigger 중 무엇인지 식별
- db_access 정책 범위 내에서 select 조회 수행
- 조회 이유를 manifest 에 기록
- 조회 결과를 cache 파일에 즉시 반영하거나 refresh 결과에 포함
- fallback 성공 시 analyze 계속 진행
- fallback 실패 시, 기존 cache 가 전혀 없으면 analyze 실패
- fallback 실패지만 기존 cache 에 부분 정보가 있으면 warning 후 진행 여부를 정책으로 정의

5. machine_spec(original) 생성 시 cache 우선 사용
- naming 보강
- mapper namespace 추정 보정
- sql id / table / code 참조 보강
- unresolved 감소에 cache 를 활용

6. analyze.manifest 에 아래 필드 추가 또는 보강
- cache.refresh_attempted
- cache.refresh_required
- cache.refresh_status
- cache.refreshed_files[]
- cache.hit_categories[]
- cache.miss_categories[]
- cache.db_fallback_used
- cache.db_fallback_reason[]
- cache.db_fallback_queries[]
- cache.db_fallback_status

중요
/analyze 본문에 “cache 에 정보가 없으면 실제 DB 접근 후 필요한 정보를 찾고, 그 결과를 cache 와 manifest 에 반영한다”는 절차가 명시적으로 들어가야 한다.

4단계. build 체인에 “cache 부족 시 DB fallback” 절차 추가

/build command 를 아래 기준으로 수정하라.

반드시 포함할 것

1. build 는 기본적으로 cache refresh 를 수행하지 않는다.
2. build 는 final machine_spec 생성/검증, naming 검증, namespace/sql id 검증, 테이블/컬럼/코드 정의 확인 시 convention cache 를 먼저 참조한다.
3. 아래 상황에서는 DB fallback 가능
- cache miss
- cache insufficient
- cache mismatch 가 명확
- machine_spec 검증에 꼭 필요한 구조 정보가 cache 로 확인되지 않음

4. DB fallback 절차
- 필요한 정보 종류 식별
- db_access 정책 범위 내 select 조회
- fallback 이유를 build.manifest 또는 verify_result 에 기록
- 조회 결과가 신뢰 가능하면 build 검증에 사용
- 필요 시 cache 파일 반영 또는 warning 기록

5. build.manifest / verify_result 에 아래 정보 반영
- cache.used
- cache.hit_categories[]
- cache.miss_categories[]
- cache.insufficient
- cache.mismatch_detected
- cache.db_fallback_used
- cache.db_fallback_reason[]
- cache.db_fallback_status
- cache_checks[]
- cache_warnings[]

6. generate skill 호출 전 검증 단계에 아래를 포함
- naming 규칙 검증 시 cache 우선
- mapper namespace / sql id 확인 시 cache 우선
- table/column/code 존재성 확인 시 cache 우선
- cache 불충분 시 DB fallback 가능

중요
/build 는 “cache 를 먼저 보고, 없으면 필요한 만큼만 DB 에 select 조회한 뒤 생성/검증을 진행한다”는 절차가 분명해야 한다.

5단계. fix 체인에 “cache 부족 시 DB fallback” 절차 추가

/fix command 와 system-fix skill 을 아래 기준으로 수정하라.

반드시 포함할 것

1. fix 분석 시 아래 유형은 cache 우선 조회
- naming 오류
- sql id / namespace 오류
- table/column/code 정의 불일치
- function/trigger 관련 구조 mismatch
- skeleton/generated 결과와 실제 구조 충돌

2. cache 에 정보가 없거나 부족하면 DB fallback 판정
- 필요한 카테고리 식별
- db_access 정책 범위 내 select 조회
- fallback 이유 기록
- 결과를 원인 분석에 반영

3. fix.manifest 에 아래 필드 반영
- cache.used
- cache.hit_categories[]
- cache.miss_categories[]
- cache.insufficient
- cache.mismatch_detected
- cache.db_fallback_used
- cache.db_fallback_reason[]
- cache.db_fallback_status

4. system-fix skill 이 아래를 활용 가능해야 한다
- cache 와 현재 산출물의 불일치 근거
- cache 와 DB 조회 결과의 차이
- policy/template/skeleton/contract 보정 필요 판단

중요
/fix 는 단순히 “cache 를 본다”가 아니라, “cache 로 충분하지 않으면 DB 조회로 보완하여 원인을 구조적으로 분류한다”는 절차가 드러나야 한다.

6단계. runtime policy 보정

- system/policies/runtime/allowed_paths.yml
- system/policies/runtime/context_isolation.yml

반드시 반영할 것

1. analyze/build/fix 각 단계의 cache 읽기 가능 경로
2. analyze 의 cache refresh 및 DB fallback 허용 범위
3. build/fix 의 기본 동작은 read-only cache 참조이며, cache 부족 시 lookup 허용
4. DB fallback 은 cache-first 이후에만 허용
5. DB fallback reason 기록 의무
6. archive 는 cache refresh source 또는 lookup source 가 될 수 없음
7. cache 관련 failure / warning / continue 정책
8. fallback 결과를 cache 에 반영할지 여부와 기준

db_access.yml 에서는 최소 아래 문장을 정책 수준으로 분명히 하라.

- cache miss 또는 cache insufficient 또는 cache mismatch 인 경우에만 lookup 허용
- lookup 결과는 manifest 에 reason 과 category 를 기록
- analyze 는 lookup 결과를 cache 보강에 사용할 수 있음
- build/fix 는 기본적으로 read-only 참조이며, cache 갱신이 필요하면 명시적으로 기록
