가장 큰 문제는 runtime 계약과 command 구현이 아직 완전히 일치하지 않는다는 점입니다. runtime.yml은 build의 시작 조건을 required_state: ["task_reviewed"]로 두고, task_analyzed 상태에서는 허용 명령이 clean뿐이며 다음 단계는 task_reviewed라고 명시합니다. 그런데 실제 /build command는 실행 전 확인에서 current.phase == task_analyzed를 요구합니다. 즉, runtime은 build 전에 user review 상태 전이를 요구하고, build command는 그 전이 없이 바로 build를 허용하고 있습니다. 이건 지금도 가장 큰 정합성 문제입니다.

두 번째 문제는 build가 호출하는 skill 구성에 구멍이 있다는 점입니다. /build command는 “Review Skill 호출 → final/machine_spec.yml 생성”을 첫 핵심 단계로 적고, review skill이 diff 분석과 final machine_spec 생성을 담당한다고 씁니다. 그런데 .claude/settings.json에 등록된 skills는 generate, deploy, system-fix뿐이고, .claude/skills 디렉터리에도 실제 파일은 deploy.md, generate.md, system-fix.md만 보입니다. 즉, build가 전제하는 review skill이 현재 저장소에는 등록도, 파일도 보이지 않습니다. 이 상태면 build 절차 설명이 실행 자산과 맞지 않습니다.

세 번째는 cache/DB fallback 반영이 문서와 command 사이에서 아직 어긋난다는 점입니다. 새 보고서는 “cache가 부족하면 실제 DB SELECT로 보완하는 절차를 analyze/build/fix 전체에 일관되게 반영했다”고 적습니다. 그런데 실제 /build command는 convention cache 검증 단계에서 **“cache에 해당 항목이 없으면 warning 기록, DB fallback은 하지 않음”**이라고 명시합니다. 즉, 보고서는 “통합 완료”를 말하지만, build command 본문은 여전히 fallback 미사용 쪽입니다. 이건 문서보다 구현 프롬프트가 뒤처져 있다는 뜻입니다
