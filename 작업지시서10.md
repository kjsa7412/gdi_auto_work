작업 목적

이번 작업은 gdi_auto_work 레포에서 /build 마지막에 deploy 단계를 선택적으로 연결하는 수정 작업이다.
현재 /build command 는 review → final machine_spec 생성 → convention cache 검증 및 필요 시 DB fallback → generate → verify_result 생성 → deliverables/report 정리 → build.manifest 완료 → active_context 갱신까지만 수행하고 종료한다.
반면 deploy skill 은 이미 별도로 존재하며, deliverables 를 실제 프로젝트 반영 대상 경로로 매핑하고 overwrite 위험을 경고한 뒤 사용자 확인 후 복사하도록 설계되어 있다.
또한 settings.json 에는 review, generate, deploy, system-fix, cache-refresh skill/command 가 이미 등록되어 있어, build 이후 deploy 를 선택적으로 연결하는 방향이 현재 구조와 잘 맞는다.

따라서 이번 작업에서는 기존 구조를 유지한 상태에서, /build 마지막에 deploy skill 을 자연스럽게 연결하되, 실제 외부 반영은 항상 선택적이며 기본값은 “배포 준비 또는 반영 대상 정리” 수준으로 동작하도록 수정하라.

현재 확인된 상태

1. /build command 에는 deploy skill 호출 단계가 없다.
2. /build 는 deliverables/report 정리 후 종료하고, 결과 보고에서 /fix 또는 /clean 만 안내한다.
3. deploy skill 은 존재하며 build 또는 fix 결과물을 입력으로 받아 반영 대상 구조 정리, 충돌 경고, 사용자 확인 후 실제 복사를 수행하도록 설계돼 있다.
4. settings.json 에는 review, generate, deploy, system-fix, cache-refresh 가 등록되어 있다.
5. 따라서 deploy 는 “없는 기능”이 아니라 “build orchestration 에 아직 연결되지 않은 기능”이다.

절대 규칙

1. 이번 작업은 build-deploy 연결 강화 작업이다. 기존 아키텍처를 다시 설계하지 말아라.
2. deploy 는 build 의 필수 성공 조건이 아니다. build 는 생성/검증/결과물 정리까지로도 완료될 수 있어야 한다.
3. deploy 는 verify_result 가 pass 또는 warn 일 때만 선택적으로 진입 가능해야 한다.
4. deploy 의 기본 동작은 “배포 준비/반영 대상 정리” 이다.
5. 실제 파일 복사는 target path 가 명확하고, overwrite 위험 검토가 끝났고, 사용자 확인이 있는 경우에만 수행하도록 유지하라.
6. archive 는 여전히 history only 이며 deploy 입력으로 사용 금지다.
7. build, deploy, runtime, manifest, report 문서가 서로 같은 규칙을 말해야 한다.
8. fix 결과물에 대한 deploy 가능성도 훼손하지 말고 유지하라.
9. deploy 연결 추가로 인해 build 가 지나치게 무거워지거나 무조건 외부 반영을 수행하는 구조가 되면 안 된다.

핵심 목표

1. /build 마지막에 deploy skill 진입 조건이 명시되어야 한다.
2. deploy 는 기본적으로 prepare only 모드로 동작해야 한다.
3. verify_result status 가 pass 또는 warn 일 때만 deploy 준비 단계로 넘어갈 수 있어야 한다.
4. build.manifest 또는 report 에 deploy 수행 여부와 준비 결과가 남아야 한다.
5. target path 미확정, 충돌 위험, 사용자 미확인 상황에서는 실제 복사 없이 준비 결과만 남겨야 한다.
6. build 와 deploy 의 책임이 섞이지 않아야 한다.

작업 순서

1단계. 현재 build 와 deploy 연결 상태 점검

아래 파일을 먼저 읽고 현재 build 와 deploy 의 연결 상태를 점검하라.

- .claude/commands/build.md
- .claude/skills/deploy.md
- .claude/settings.json
- system/config/runtime.yml
- system/policies/runtime/allowed_paths.yml
- system/policies/runtime/context_isolation.yml
- system/schemas/build_manifest.schema.json
- system/schemas/verify_result.schema.json
- work/task/2.Working/manifests/build.manifest.yml
- work/task/2.Working/manifests/verify_result.yml
- command_implementation_report.md
- manifest_contract_report.md

반드시 정리할 것

1. build 종료 시점이 어디인지
2. deploy skill 입력/출력이 무엇인지
3. deploy 가 build 이후 자동 호출 가능한 구조인지
4. runtime/policy/schema 가 deploy 를 build 의 후속 단계로 허용하는지
5. build 결과물과 deploy 입력 경로가 일치하는지
6. build 성공과 deploy 성공을 어떻게 분리할지

이 단계 결과를 “build-deploy 연결 누락 목록”으로 정리하라.

2단계. build 와 deploy 책임 분리 재정의

아래 원칙을 기준으로 책임을 정리하라.

build 책임
- review skill 호출
- final machine_spec 생성
- cache 검증 및 필요 시 DB fallback
- generate skill 호출
- verify_result 생성
- deliverables/report 정리
- build.manifest 완료
- deploy 가능 여부 판정
- 선택적 deploy 준비 호출

deploy 책임
- deliverables 수집
- target path 계산
- overwrite/conflict 검토
- prepare only 결과 정리
- 사용자 확인 시 실제 복사
- deploy summary 생성

중요
build 는 생성 시스템이고, deploy 는 반영 준비 또는 반영 단계이다.
따라서 build 성공 여부와 deploy 성공 여부는 별도 상태로 구분 가능해야 한다.

3단계. /build command 에 deploy 호출 단계 추가

/build command 를 아래 기준으로 수정하라.

반드시 포함할 것

1. verify_result 생성 후 deploy 진입 판정 단계 추가
2. deploy 진입 조건
- verify_result.status 가 pass 또는 warn
- deliverables 경로가 존재
- deploy 금지 상태가 아님
3. deploy 호출 방식
- 기본값은 prepare only
- target path 가 framework_manifest 또는 사용자 입력으로 식별 가능한 경우 deploy skill 호출
- 실제 복사는 사용자 확인 전까지 수행하지 않음
4. deploy 수행 후 build.manifest 또는 report 에 아래 정보 기록
- deploy.attempted
- deploy.mode
- deploy.status
- deploy.summary_path 또는 deploy.summary
- deploy.warnings[]
- deploy.conflicts[]
5. deploy 실패 처리
- prepare only 단계 실패는 build 결과물 자체를 무효화하지 않음
- 단, warning 또는 partial 상태로 기록
- 생성물 자체가 없으면 build 실패 유지
6. 결과 보고 보정
- deploy 준비 완료 시 그 경로와 대상 파일 수 안내
- target path 미확정이면 “배포 준비까지만 완료”로 안내
- 실제 복사 미수행이면 그 사실을 명확히 표시

권장 절차 예시

1. Deliverables/Report 정리
2. deploy 필요 여부 판정
3. 필요 시 deploy skill 호출 (prepare only)
4. deploy summary 반영
5. build.manifest 완료
6. active_context 갱신
7. 결과 보고

중요
/build 마지막에 deploy 를 붙이되, build 성공 조건은 여전히 생성/verify/deliverables 완료여야 한다.
deploy 준비 실패가 build 전체 실패가 되지 않도록 경계를 명확히 하라.

4단계. deploy skill 보정

deploy skill 을 아래 기준으로 보정하라.

반드시 포함할 것

1. build 에서 호출될 때 기본 모드가 prepare-only 임을 명시
2. 입력
- work/task/3.Result/deliverables/**
- 또는 work/fix/3.Result/deliverables/**
- framework_manifest 의 source roots 또는 사용자 지정 target 정보
- mode: prepare-only 또는 apply
3. prepare-only 절차
- deliverables 존재 검증
- 파일별 target path 계산
- overwrite/conflict 탐지
- summary 생성
- 실제 복사는 하지 않음
4. apply 절차
- 사용자 확인이 있는 경우에만 수행
- 복사 결과 기록
5. 출력
- deploy summary
- prepared_paths
- copied_paths
- warnings
- conflicts
- mode
- status
6. 실패 조건 보정
- deliverables 없음
- target path 미확정
- path 계산 불가
- conflict 해결 불가
7. build 와의 연계 고려
- prepare-only 결과는 build.report/build.manifest 에 바로 붙일 수 있어야 함

중요
deploy skill 은 독립적으로도 쓸 수 있어야 하지만, 이번 수정에서는 build 마지막 후속 단계로 호출될 수 있게 인터페이스를 분명히 하라.

5단계. runtime/policy 보정

아래 파일을 수정하라.

- system/config/runtime.yml
- system/policies/runtime/allowed_paths.yml
- system/policies/runtime/context_isolation.yml

반드시 반영할 것

1. build 단계에서 deploy skill 호출 허용
2. deploy 가 읽을 수 있는 경로
- work/task/3.Result/deliverables/**
- work/fix/3.Result/deliverables/**
- system/config/**
- framework manifest
3. deploy 가 쓸 수 있는 경로
- work/task/3.Result/report/**
- 또는 deploy summary 저장 위치
- 실제 복사 시 target path
4. archive 입력 금지 유지
5. deploy prepare-only 와 apply 구분
6. build 이후 deploy 준비가 선택적 후속 단계임을 명시
7. apply 는 사용자 확인 후만 허용된다는 규칙 반영

필요 시 deploy_policy.yml 을 추가해도 된다.

7단계. 샘플 manifest 갱신

아래 샘플을 실제 수정 규칙에 맞게 갱신하라.

- work/task/2.Working/manifests/build.manifest.yml
- work/task/2.Working/manifests/verify_result.yml

샘플에는 최소 아래 사례가 드러나야 한다.

1. build 성공 후 deploy prepare-only 수행
2. target path 계산 성공
3. overwrite conflict 없음 또는 warning 수준 conflict 존재
4. 실제 복사는 아직 미수행
5. build 는 completed, deploy 는 prepared 또는 warn 상태

8단계. settings.json 보정

.claude/settings.json 을 보정하라.

반드시 반영할 것

1. 현재 등록된 deploy skill 을 build 후속 단계로 설명
2. workflow.flow 설명에 deploy 준비 단계를 반영
예시
analyze → (사용자 검토) → build [review → generate → optional deploy-prepare] → (fix if needed) → clean
3. 권한/원칙과 충돌하지 않게 유지
4. cache-refresh, review, deploy 등 현재 등록 자산과 실제 command/skill 파일이 일치하는지 확인

9단계. 문서 보정

아래 문서를 수정하라.

- command_implementation_report.md
- manifest_contract_report.md

반드시 반영할 것

1. build 마지막에 선택적 deploy prepare 단계가 추가되었다는 점
2. build 성공과 deploy 준비 성공은 구분되는 점
3. deploy apply 는 별도 사용자 확인이 필요하다는 점
4. build.manifest / verify_result 확장 내용
5. target path 미확정 시 prepare-only 로 끝나는 정책

10단계. 최종 보고서 작성

build_deploy_integration_report.md 를 작성하라.

반드시 아래 목차를 포함하라.

1. 작업 개요
2. 현재 문제 인식
3. build 와 deploy 역할 재정의
4. /build 보강 내용
5. deploy skill 보정 내용
6. runtime/policy 보정 내용
7. schema/manifest 확장 내용
8. 샘플 manifest 갱신 결과
9. 남은 한계 및 미해결 항목
10. 다음 단계 권장 작업

가능하면 어떤 파일의 어떤 항목이 수정되었는지 경로 기준으로 명시하라.

출력 방식

1. 실제 파일 생성이 가능하면 지정된 경로에 파일을 생성하라.
2. 실제 파일 생성이 불가능하면 파일별 내용을 구분해서 전부 출력하라.
3. 보고서만 쓰고 끝내지 말고, command/policy/schema/manifest 샘플 수정본까지 함께 작성하라.
4. 미확정 항목은 “미확정” 또는 “근거 부족”으로 표시하라.
5. 단계 종료 시 작성 또는 수정한 파일 목록을 반드시 정리하라.

최종 완료 기준

1. /build 마지막에 deploy 진입 판정 및 deploy skill 호출 절차가 포함되어 있어야 한다.
2. deploy skill 이 prepare-only 기본 모드로 build 와 자연스럽게 연결되어 있어야 한다.
3. build 성공과 deploy 준비 성공이 구분되어 있어야 한다.
4. build.manifest 와 verify_result 에 deploy 관련 필드가 추가되어 있어야 한다.
5. settings.json workflow 설명이 실제 구조와 일치해야 한다.
6. report 문서에 build-deploy 연결 내용이 반영되어 있어야 한다.
7. build_deploy_integration_report.md 가 작성되어 있어야 한다.

작업 시작 전 선언

이번 작업은 “build 마지막 deploy 연결 통합 수정 단계”이다.
따라서 새로운 아키텍처를 다시 설계하지 말고, 기존 구조를 유지한 상태에서 /build 마지막에 deploy skill 을 선택적으로 연결하고, 기본은 prepare-only 로 동작하도록 수정하라.
