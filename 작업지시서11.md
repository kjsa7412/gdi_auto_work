아래 내용을 잘 정리해서 readme.md 아래에 첨부하라    


이 시스템은 그냥 “프롬프트 모음”이 아니라, AI 시스템 관점에서 보면 통제형 에이전트 파이프라인에 가깝습니다. 핵심 특성은 다음 다섯 가지로 요약할 수 있습니다.

자율형 시스템이 아니라 통제형 시스템
대화형 추론이 아니라 manifest 기반 단계 실행 시스템
문맥 누적형이 아니라 context 격리형 시스템
1회 생성형이 아니라 수정·보정 루프를 내장한 시스템
AI가 전부 결정하는 구조가 아니라 사람 검토를 끼워 넣은 half-automated 구조

이 해석은 레포가 single-active workspace, manifest/schema 검증 우선, skeleton 기반 생성, archive 입력 금지를 원칙으로 두고, workflow를 analyze → (사용자 검토) → build → (fix if needed) → clean으로 정의한 점에서 직접 읽힙니다.

1. 통제전략: “자율성 최소화, 결정 가능성 최대화”

이 시스템의 가장 큰 특징은 AI의 자유도를 의도적으로 낮춘다는 점입니다.
보통 에이전트 시스템은 “알아서 읽고, 알아서 판단하고, 알아서 만들게” 하다가 문맥 오염이나 구조 붕괴가 생기는데, 이 시스템은 반대로 갑니다.

구체적으로는:

동시에 하나의 task 또는 fix만 허용하는 single-active workspace
입력/출력은 반드시 schema와 manifest를 통과해야 하는 계약 우선
자유 생성 금지, skeleton 기반 생성 강제
archive를 실행 입력으로 쓰지 않는 과거 분리
build도 곧바로 반영하지 않고 verify와 선택적 deploy 준비를 거치는 단계화

이건 AI를 “똑똑한 창작자”로 보는 게 아니라, 정해진 공정 안에서 움직이는 작업 엔진으로 보는 전략입니다. 레포 설정에도 이 원칙이 직접 적혀 있고, command 구현 보고서도 command = orchestration, skill = 세부 실행으로 역할을 분리합니다.

즉 이 시스템의 통제전략은 한마디로
**“AI를 믿지 않아서 막는 게 아니라, AI가 잘할 수 있는 범위만 남기고 나머지는 제약으로 고정하는 방식”**입니다.

2. Manifest 중심성: “프롬프트 시스템이 아니라 실행 계약 시스템”

이 시스템에서 manifest는 부수 파일이 아니라 중심 개념입니다.
manifest_contract_report.md는 manifest를 analyze/build/fix/clean 흐름의 입출력, 상태전이, 검증 규칙을 기계적으로 검증 가능한 계약층이라고 정의합니다.

이게 왜 중요하냐면, 일반적인 AI 작업은 “모델이 직전 대화 문맥을 기억하니까” 이어서 합니다.
그런데 이 시스템은 그렇게 하지 않습니다. 대신 각 단계가 끝날 때마다

무엇을 입력으로 썼는지
무엇을 출력으로 만들었는지
어떤 화면이 추출됐는지
어떤 cache를 썼는지
DB fallback이 있었는지
어떤 상태로 끝났는지

를 manifest에 남깁니다. analyze/build/fix manifest 샘플과 schema가 별도로 존재하는 이유가 바로 이것입니다.

즉 이 시스템은 “대화가 다음 단계를 이어준다”가 아니라
**“manifest가 다음 단계를 이어준다”**는 구조입니다.

이건 AI 시스템 설계에서 굉장히 중요한 특성입니다.
왜냐하면 실행을 대화 메모리에서 파일 기반 상태 데이터로 옮겨 놓았기 때문입니다.

3. 컨텍스트 오염 방지 전략: “문맥을 쌓지 않고, 문맥을 격리한다”

사용자가 말한 “컨텍스트 오염” 관점에서 보면, 이 시스템은 매우 의식적으로 오염을 막는 방향입니다.

3-1. single-active

한 번에 하나의 task 또는 fix만 다룹니다.
여러 작업이 동시에 열리면 AI는 이전 작업의 화면, 규칙, spec, 수정 이력을 섞기 쉽습니다. 이 시스템은 그걸 구조적으로 막습니다.

3-2. archive 입력 금지

과거 작업은 남겨 두되, 실행 입력으로 쓰지 못하게 합니다.
즉 “기록은 보존하지만 문맥은 차단”합니다. 이건 흔한 로그 보존과는 다릅니다. 보통은 로그를 남기면 다시 참조하고 싶어지는데, 이 시스템은 오히려 기록 보존과 실행 입력을 분리합니다.

3-3. work 영역 분리

work/task, work/fix, archive를 나눈 이유도 같습니다.
실행 중인 맥락과 완료된 맥락을 섞지 않으려는 것입니다. 저장소 구조도 .claude, system, work로 나뉘고, work 안에 manifest와 active context가 따로 있습니다.

3-4. spec 이원화

people_spec과 machine_spec을 나누는 것도 컨텍스트 오염 방지 전략입니다.
사람이 보는 문서와 시스템이 읽는 문서를 분리하면, 사람이 가독성을 위해 넣은 설명 문장이 기계 입력을 오염시키지 않습니다. manifest 계약 보고서도 people_spec은 사람 검토용 Markdown, machine_spec은 build 직접 입력용 YAML로 나눕니다.

즉 이 시스템은 문맥을 풍부하게 모으는 게 아니라,
문맥을 용도별로 잘라서 오염 없이 전달하는 구조입니다.

4. 루프 구조: “선형 파이프라인이 아니라 제한된 피드백 루프”

겉으로 보면 analyze → build → fix → clean이니까 선형처럼 보이지만, 실제로는 제한된 루프 시스템입니다.

4-1. 사용자 검토 루프

analyze 후에 바로 build하지 않고 사용자가 final people_spec을 검토하게 둡니다.
즉 AI가 1차 구조화를 하고, 사람이 의미 보정을 한 뒤 다음 단계로 갑니다. runtime도 task_analyzed 상태 설명에 “사용자가 final people_spec을 검토/수정 후 /build 실행”을 명시합니다.

이건 AI 시스템 관점에서 보면 human-in-the-loop 루프입니다.

4-2. fix 루프

build가 끝나도 시스템은 종료되지 않습니다.
fix는 단순 패치가 아니라 영향 범위를 판정하고, 필요하면 재분석/재생성/정책 보정을 트리거합니다. command 구현 보고서도 fix는 system-fix skill과 연결되고, 반복 오류일 경우 시스템 차원 보완 대상으로 본다고 설명합니다.

즉 이 시스템은 “실패하면 끝”이 아니라
실패를 다음 규칙 개선의 입력으로 환원하는 구조입니다.

4-3. deploy 루프

지금은 build 마지막에 deploy prepare를 붙이는 쪽으로 정리 중인데, 그 철학도 유사합니다.
build 성공과 deploy 성공을 분리해, 생성과 반영을 같은 단계로 뭉개지 않습니다. command 구현 보고서도 build의 선택적 deploy prepare를 별도 책임으로 둡니다.

즉 루프가 무한정 자유로운 건 아니고,
허용된 피드백 루프만 갖는 제어형 루프 시스템입니다.

5. 지식 접근 전략: “LLM 기억보다 외부 근거 우선”

이 시스템은 AI가 알아서 기억하는 방식보다 외부 근거 참조를 우선합니다.

대표적인 게 system/cache/convention/과 DB fallback 구조입니다.
설계상 기본 모드는 cache-first이고, 부족할 때만 lookup을 허용합니다. 사용자가 최근 보강한 방향도 cache가 없으면 DB SELECT로 보완하고, 그 결과를 manifest에 남기는 방식입니다. 이는 “모델의 추정”보다 “확인 가능한 정의”를 우선하는 전략입니다.

즉 이 시스템은 LLM이 “대충 맞게 상상하는” 걸 싫어합니다.
대신

PPT
people_spec
machine_spec
convention cache
정책 파일
skeleton
manifest
필요 시 DB lookup

같이 외부화된 근거 레이어를 여러 겹 쌓아 둡니다.

이건 AI 시스템 관점에서 보면
내재 기억형이 아니라 외재 근거형 시스템입니다.

6. 생성 전략: “자유 생성이 아니라 constrained generation”

이 시스템은 생성 자체를 없애진 않지만, 생성 자유도를 꽤 줄입니다.

screen-type으로 유형 먼저 고정
template_selection으로 skeleton 조합 결정
skeleton_contract로 placeholder 계약 강제
generate skill은 skeleton 기반 치환만 수행
framework policy 위반 시 실패 처리

command 구현 보고서도 generate skill의 절대 규칙을 “skeleton 기반만 허용, 자유 생성 금지”로 명시합니다.

이건 AI 시스템 설계에서 아주 전형적인 constrained generation입니다.
즉 모델이 전체 결과를 창작하는 게 아니라,
정해진 골격 안에 값만 채우는 방식입니다.

장점은 품질 일관성과 재현성이 좋아진다는 점이고,
단점은 예외 케이스에 약하고, skeleton이 부족하면 생성력이 급격히 떨어진다는 점입니다.

7. 상태 기계 특성: “에이전트라기보다 workflow engine에 가까움”

runtime을 보면 이 시스템은 사실상 workflow engine입니다.
idle, task_prepared, task_analyzed, task_built, fix_applied, failed, cleaned 같은 상태가 있고, 각 상태마다 허용 명령과 다음 상태가 정의됩니다. 즉 자연어 기반 자율 agent보다는 상태 전이형 실행기에 가깝습니다.

이 관점에서 보면 command는 “도구”가 아니라 상태 전이 트리거입니다.

analyze = idle/task_prepared → task_analyzed
build = task_analyzed → task_built
fix = task_built → fix_applied
clean = 여러 상태 → idle/cleaned

이런 식입니다.

그래서 이 시스템은 일반적인 “agentic coding system”보다
stateful workflow + AI generation modules에 더 가깝습니다.

8. 인간 개입 전략: “AI 단독 결정 금지 구간이 있다”

이 시스템은 일부러 사람이 개입해야 하는 지점을 남겨둡니다.

대표적으로:

analyze 후 people_spec 검토
deploy apply 전 사용자 확인
fix 이후 정책 보정 여부 판단
target path 미확정 시 prepare-only로 멈춤

command 구현 보고서는 deploy도 기본 모드를 prepare-only로 두고, 실제 복사는 사용자 확인 후에만 하도록 설계했다고 적습니다.

이건 중요한 특성입니다.
왜냐하면 이 시스템은 “AI 자동화율 100%”를 목표로 하지 않기 때문입니다.
대신 위험이 큰 결정 구간만 인간 승인 지점으로 남겨 둔 반자동 시스템입니다.

9. 이 시스템의 강점과 약점
강점
문맥 오염에 강함
재현성이 좋음
실행 이력 추적이 쉬움
정책/계약 위반을 구조적으로 발견 가능
사람 검토를 끼워 품질 손실을 줄임

이건 모두 manifest 중심성, single-active, skeleton 강제, state machine 구조에서 나옵니다.

약점
초기 설정이 복잡함
schema/runtime/command 불일치가 생기면 전체가 막힘
예외 케이스 처리 유연성이 낮음
skeleton과 정책 품질에 지나치게 의존함
“간단한 작업인데도 공정이 무겁다”는 느낌이 생길 수 있음

즉 이건 유연성보다 통제성을 택한 시스템입니다.

10. 한 줄씩 개념으로 정의하면

이 시스템을 AI 시스템 개념으로 번역하면 이렇게 볼 수 있습니다.

통제전략: policy-first, schema-first, stateful control
Manifest: 실행 계약이자 단계별 상태 증거물
Context 관리: single-active + archive 차단 + spec 분리 기반의 context isolation
생성 방식: free-form generation이 아니라 constrained generation
루프 구조: user review loop + fix loop + optional deploy loop
지식 접근: model memory보다 externalized evidence 우선
운영 철학: autonomous agent보다 controlled workflow engine
11. 결론

이 시스템은 본질적으로
“AI가 알아서 다 하는 시스템”이 아니라, AI를 통제 가능한 공정 안에 넣어 다루는 시스템입니다.

즉 성격상:

LLM-native chat system 보다는
manifest-driven workflow system 에 가깝고,
autonomous agent 보다는
constrained orchestration engine 에 가깝습니다.
