작업 목적
본 작업은 새 시스템 구축을 위한 “프레임워크 분석 단계”만 수행하는 것이다.
아직 명령어 구현, 코드 생성, 배포 자동화, build/fix 로직 구현을 하지 말고, 오직 프레임워크 분석 결과를 기준으로 system/config, system/policies/framework, system/cache/convention, system/templates/screen-types 를 채우기 위한 근거를 수집·정리·정의하는 데 집중하라.
기반 시스템의 구조나 파일을 그대로 복제하지 말고, 제공된 실제 프레임워크와 레퍼런스 소스를 분석하여 새 시스템이 따라야 할 규칙과 골격만 추출하라.

절대 규칙

추정으로 규칙을 만들지 말고 반드시 실제 파일 구조, 실제 클래스, 실제 xml, 실제 html을 근거로 분석하라.
프레임워크에서 공식적으로 강제되는 구조와, 프로젝트에서 관행적으로 사용하는 구조를 구분하라.
예외 구현 사례를 표준으로 승격하지 말아라. 반복적으로 나타나는 공통 패턴만 표준 후보로 채택하라.
“가능할 것 같다” 수준의 표현을 금지한다. 근거 파일 경로와 함께 판단하라.
현재 단계에서는 skeleton 파일 자체를 대량 생성하지 말고, skeleton 추출이 가능하도록 패턴과 계약을 먼저 정의하라.
controller, service, xml, html 간 연결 구조를 반드시 분석하라.
화면 1개 단위로 파일이 어떻게 연결되는지 실제 예시를 최소 3건 이상 확보하라.
분석 결과는 반드시 새 시스템의 아래 파일을 채우기 위한 형태로 정리하라.
아직 analyze/build/fix command 내용은 작성하지 말고, 프레임워크 분석 결과물만 작성하라.
결과는 사람이 검토 가능한 보고서와, 시스템이 그대로 읽을 수 있는 yml 초안 형태 둘 다 만들어라.

출력 방식

실제 파일 생성이 가능하면 지정된 경로에 파일을 생성하라.
실제 파일 생성이 불가능하면, 파일별 내용을 구분해서 전체 내용을 한 번에 출력하라.
보고서만 쓰고 끝내지 말고, 반드시 yml/sql 초안까지 함께 작성하라.
불확실한 항목은 “추정”이라고 숨기지 말고 “미확정” 또는 “근거 부족”으로 명시하라.
단계가 끝날 때마다 어떤 파일이 작성되었는지 목록으로 정리하라.

최종 완료 기준

새 시스템이 프레임워크 분석 결과만으로도 화면유형 분류와 skeleton 선택의 기준을 가질 수 있어야 한다.
naming, controller, service, xml, html 규칙이 각각 별도 정책 파일로 정리되어 있어야 한다.
금지 패턴이 정의되어 있어야 한다.
framework_manifest.yml 이 작성되어 있어야 한다.
screen-types/*.yml 이 작성되어 있어야 한다.
convention cache 수집용 sql 초안이 작성되어 있어야 한다.
framework_analysis_report.md 에 실제 근거 파일 경로가 포함되어 있어야 한다.

작업 시작 전 선언
이번 작업은 “프레임워크 분석 및 정책/계약 초안 작성” 단계이며, 실제 코드 생성 단계가 아니다.
따라서 명령어 구현, skeleton 대량 생성, build/fix 자동화 작성은 수행하지 말고, 그 다음 단계에서 바로 사용할 수 있는 분석 결과물과 정책 초안만 작성하라.
