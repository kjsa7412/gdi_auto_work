작업 목적
이번 작업은 “프레임워크 분석 결과를 바탕으로 실제 skeleton 템플릿을 추출·정규화하는 단계”이다.
이전 단계에서 작성된 framework 분석 결과물과 실제 프레임워크/레퍼런스 소스를 근거로, 새 시스템이 사용할 수 있는 html/xml/controller/service skeleton 파일을 생성하고, skeleton 선택 규칙과 placeholder 계약을 실제 파일 수준으로 구체화하라.
이번 단계에서는 아직 analyze/build/fix command 구현을 하지 말고, 오직 skeleton 템플릿과 그에 연결되는 정책/매핑/보고서만 작성하라.

절대 규칙

skeleton은 반드시 실제 반복 패턴을 근거로 만들어라. 임의로 예쁜 구조를 새로 설계하지 말아라.
skeleton은 “업무 로직이 없는 재사용 골격”이어야 한다. 특정 업무 화면의 상세 문구, 특정 컬럼, 특정 쿼리 내용은 일반화해서 placeholder로 치환하라.
skeleton은 프레임워크에 맞는 실제 구현 형태를 따라야 한다. 순수 html/css/js 자유 구조를 만들지 말아라.
이전 단계에서 정의한 naming, pattern, forbidden, screen-type, skeleton_contract를 우선 기준으로 삼고, 부족한 경우에만 실제 소스를 다시 확인하여 보완하라.
반복되지 않는 예외 사례를 skeleton 표준으로 채택하지 말아라.
skeleton 파일은 “바로 generate에 투입 가능한 최소 골격” 수준으로 만들어라.
공통부와 가변부를 명확히 분리하라.
placeholder 명명은 반드시 일관되게 유지하라.
skeleton 파일을 만든 뒤에는 template_selection, skeleton_contract, screen-types 정의와 충돌이 없는지 역검증하라.
아직 manifest schema 작성이나 command 구현은 하지 말고, skeleton 자산과 관련 정책 보정만 수행하라.

최종 완료 기준

list, list-detail, form, popup 4개 유형에 대해 html/xml/controller/service skeleton 파일이 존재해야 한다.
skeleton 파일은 실제 반복 패턴 기반 골격이어야 하며, 업무 특화 내용은 placeholder로 치환되어야 한다.
template_selection.yml 이 실제 skeleton 파일과 연결되어 있어야 한다.
skeleton_contract.yml 이 실제 placeholder 계약 수준으로 보정되어 있어야 한다.
screen-types/*.yml 이 skeleton 선택 기준서로 보정되어 있어야 한다.
skeleton_extraction_report.md 에 대표 샘플 경로와 설계 근거가 포함되어 있어야 한다.

작업 시작 전 선언
이번 작업은 “skeleton 추출 및 템플릿 정규화 단계”이며, 실제 명령어 구현 단계가 아니다.
따라서 analyze/build/fix command 구현, manifest schema 작성, 배포 로직 작성은 수행하지 말고, 다음 단계에서 바로 사용할 수 있는 skeleton 자산과 정책 보정본만 작성하라.
